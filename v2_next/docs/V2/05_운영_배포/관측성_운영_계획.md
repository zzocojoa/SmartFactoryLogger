# V2.0 Observability Plan (운영 모니터링 전략)

`작성일: 2025-12-31` `작성자: Smart Factory Logger Dev Team`

## 1. 개요

V2.0 웹 기술 전환 시, 기존 데스크톱 앱(V1)보다 복잡한
아키텍처(백엔드+프론트엔드+브라우저)를 가지므로, 장애 감지 및 원인 분석을 위한
체계적인 모니터링(Observability) 전략이 필수적입니다.

## 2. Health Check (상태 점검)

시스템의 생존 여부를 주기적으로 확인합니다.

### 2.1. 백엔드 API Check

- **컴포넌트**: FastAPI Server
- **Endpoint**: `GET /health` (이미 구현됨)
- **주기**: 5초
- **프론트엔드 Action**:
  - 성공(200 OK): Status Badge를 'Running' (초록색)으로 표시.
  - 실패(Timeout/500): Status Badge를 'Offline' (빨간색)으로 즉시 변경하고
    사용자에게 Toast 알림.

### 2.2. External Device Check (PLC/Sensor)

- **Watchdog Logic**: 백엔드에서 PLC 데이터 수신 timestamp가 3초 이상 갱신되지
  않으면 'Degraded' 상태로 전파.
- **프론트엔드 표시**: "Running" 상태에서도 "Last Update" 시간이 흐르지 않으면
  경고 표시.

## 3. Logging Strategy (로깅 전략)

### 3.1. 백엔드 Logs (`system.log`)

- **Library**: Python `logging` 모듈 사용 (RotatingFileHandler 적용).
- **내용**:
  - API 요청/응답 시간 (Latency)
  - PLC 통신 에러 (Connection Refused, Timeout)
  - 데이터 파싱 에러 (Pydantic Validation Error)
  - 설정 변경 적용 결과(applied/pending) 기록
- **위치**: `v2_next/backend/logs/system.log`

### 3.2. 프론트엔드 Logs (브라우저 Console)

- 일반적인 운영 로그는 콘솔에 남기되, **Critical Error**(네트워크 단절 등)는
  백엔드로 전송하지 않고(네트워크가 끊겼으므로) Local Storage에 최근 50개 에러를
  저장하여 사후 분석.
- 관측성 내보내기 시 프론트 오류 큐를 함께 포함하여 백업 가능.

## 4. Metrics (성능 지표)

초기 단계에서는 상세한 시계열 DB(Prometheus) 대신 가벼운 지표를 파일로 남깁니다.

- **API Latency**: `/stats` 엔드포인트에서 평균 응답 속도 제공.
- **Uptime**: 시스템 가동 시간 기록.
- **Export Snapshot**: 최근 내보내기 경로를 저장해 재시작 후에도 추적 가능.
