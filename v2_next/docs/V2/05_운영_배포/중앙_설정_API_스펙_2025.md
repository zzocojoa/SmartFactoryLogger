# 중앙 설정 API 스펙 (1단계 설계)

## 1. 목적
- 현장 PC(1대)가 중앙에서 최신 설정을 안전하게 받아 적용한다.
- 변경 이력과 승인 흐름을 명확히 한다.
- 강제 적용(Force)과 롤백이 가능하도록 한다.

## 2. 기본 원칙
- 설정은 **버전(version) 기반**으로 관리한다.
- 현장 PC는 **마지막 정상 버전(LKG)**을 유지한다.
- 중앙은 **RBAC + 감사 로그**를 기본 제공한다.

## 3. 인증/권한
- 인증: API Key 또는 JWT
- 권한(RBAC)
  - Admin: 전체 변경/강제 적용/롤백
  - Operator: 변경 요청(승인 필요)
  - Viewer: 읽기 전용
- 환경 변수 키
  - `SFL_CENTRAL_ADMIN_KEY`: 관리자 키
  - `SFL_CENTRAL_OPERATOR_KEY`: 운영자 키
  - `SFL_CENTRAL_VIEWER_KEY`: 조회 전용 키

## 4. 엔드포인트 (필수)

### 4.1 설정 조회 (현장 Pull)
`GET /api/config/latest?device_id={id}`
- 기능: 현재 배포 대상 최신 설정을 조회한다.
- 헤더:
  - `If-None-Match`: 마지막 버전(ETag)
- 응답:
  - 200: 최신 설정 제공
  - 304: 변경 없음

Response 예시:
```json
{
  "device_id": "LINE-01",
  "version": "2025.01.02-0012",
  "updated_at": "2025-01-02T10:20:30Z",
  "requires_restart": true,
  "config": {
    "extruder": {"ip": "192.168.10.10", "port": 12289},
    "ls_plc": {"ip": "192.168.10.220", "port": 2004},
    "spot": {"ip": "10.1.10.50", "refresh_interval": 0.2},
    "settings": {"logpath": "Z:\\logs", "snapshotpath": "Z:\\snapshots", "autosave": true},
    "logging": {"rotation_mode": "BILLET", "cycle_idle_time": 30, "cycle_threshold_press": 20}
  }
}
```

### 4.2 적용 결과 Ack
`POST /api/config/ack`
- 기능: 현장 PC가 적용 결과를 중앙에 보고한다.
- Request:
```json
{
  "device_id": "LINE-01",
  "version": "2025.01.02-0012",
  "status": "APPLIED",
  "applied_at": "2025-01-02T10:21:15Z",
  "message": "Applied and running"
}
```
- status: `APPLIED`, `FAILED`, `ROLLED_BACK`

### 4.3 설정 업데이트 (관리자)
`POST /api/config/update`
- 기능: 새로운 설정 버전을 등록한다.
- Request:
```json
{
  "device_id": "LINE-01",
  "author": "admin",
  "force": false,
  "requires_restart": true,
  "config": { ... }
}
```
- Response:
```json
{
  "version": "2025.01.02-0012",
  "status": "CREATED"
}
```

### 4.4 설정 히스토리 조회
`GET /api/config/history?device_id={id}&limit=20`
- 기능: 변경 이력 및 버전 목록 조회

### 4.5 롤백
`POST /api/config/rollback`
- 기능: 지정 버전으로 되돌림
- Request:
```json
{
  "device_id": "LINE-01",
  "version": "2025.01.01-0008",
  "reason": "Stability issue"
}
```

### 4.6 변경 요청(승인 필요)
`POST /api/config/requests`
- 기능: Operator가 변경 요청을 등록한다(승인 대기).
- Request:
```json
{
  "device_id": "LINE-01",
  "author": "operator",
  "reason": "Line tune",
  "force": false,
  "requires_restart": false,
  "config": { ... }
}
```
- Response: `{"status": "PENDING", "request_id": "REQ-00001"}`

### 4.7 요청 목록 조회
`GET /api/config/requests?device_id=LINE-01&status=PENDING`
- 기능: 요청 목록 조회(상태/디바이스 필터).

### 4.8 요청 승인/거절
`POST /api/config/requests/approve`
```json
{"request_id": "REQ-00001", "approver": "admin"}
```

`POST /api/config/requests/reject`
```json
{"request_id": "REQ-00001", "rejector": "admin", "comment": "invalid"}
```

### 4.9 정책 조회/수정
`GET /api/config/policy`
`POST /api/config/policy`
```json
{"approval_required_for_operator": true, "auto_approve_admin": true}
```

### 4.10 간이 관리 UI
`GET /ui`
- 최신 설정/요청을 HTML로 확인.

## 5. Force 적용 정책
- `force=true`인 경우 로컬 오버라이드를 무시하고 덮어쓴다.
- 현장 PC는 **사용자 확인 없이 즉시 적용**한다.
- 강제 적용은 Admin만 가능.

## 6. 버전 규칙
- 형식: `YYYY.MM.DD-XXXX` (일자 + 시퀀스)
- 해시/서명 옵션: 변경 위변조 방지

## 7. 에러 코드
- 400: 입력 오류/버전 충돌
- 401/403: 권한 오류
- 409: 동시 업데이트 충돌
- 500: 서버 오류

## 8. 최소 동작 시나리오
1) 현장 PC가 `/latest` 호출
2) 버전 변경 감지 → 적용
3) 결과를 `/ack`으로 보고
4) 중앙은 성공/실패 기록

## 9. 확장 고려사항
- 설정 서명(Signature) 추가
- 장비 상태/헬스 연동
- 다중 장비 그룹 배포

## 10. 구현 메모(현행)
- 서버 엔트리: `v2_next/backend/central_server.py`
- 기본 포트: 9000 (`python -m backend.central_server`)
- 저장소 모드: `SFL_CENTRAL_STORE_MODE=json|sqlite`
  - 기본 경로: `SFL_CENTRAL_STORE_DIR` (미설정 시 `APPDATA\\SmartFactoryLogger`)
  - JSON: `SFL_CENTRAL_STORE_PATH` 또는 `APPDATA\\SmartFactoryLogger\\central_config_store.json`
  - SQLite: `SFL_CENTRAL_DB_PATH` 또는 `APPDATA\\SmartFactoryLogger\\central_config_store.db`
- 감사 로그: `APPDATA\\SmartFactoryLogger\\central_audit.log`
- 인증 키: `SFL_CENTRAL_API_KEY` 또는 `SFL_CONFIG_API_KEY`
- 관리자 키: `SFL_CENTRAL_ADMIN_KEY` (없으면 API 키와 동일)
- 운영자/뷰어 키: `SFL_CENTRAL_OPERATOR_KEY`, `SFL_CENTRAL_VIEWER_KEY`
- 간이 UI 액션: `/ui`, `/ui/approve`, `/ui/reject`, `/ui/policy`
- 간이 UI 기능: 업데이트 생성/요청 생성/정책 변경/승인·거절 (토큰 필요)
