# 동기화 에이전트 구현 플로우 (3단계)

## 1. 목적
- 중앙 설정을 주기적으로 동기화하고 안정적으로 적용한다.
- 네트워크 장애 시에도 로컬 캐시로 안전하게 운영한다.

## 2. 동작 주기
- 기본 주기: 30~60초
- 실패 시 백오프: 1분 → 2분 → 5분 → 10분
- 최대 백오프 도달 시에도 최소 10분마다 재시도

## 3. 기본 흐름
1) 현재 로컬 버전 확인(`config_meta.json`)
2) 중앙 최신 버전 조회(`/api/config/latest`)
3) 변경 없음(304) → 다음 주기로 대기
4) 변경 있음 → 다운로드 → 검증 → 적용
5) 적용 결과를 `/api/config/ack`로 보고

## 4. 로컬 캐시 구조
- `config.ini` : 실제 적용 파일
- `config_meta.json` : 버전/동기화 상태 기록
```json
{
  "device_id": "LINE-01",
  "version": "2025.01.02-0012",
  "last_sync": "2025-01-02T10:21:15Z",
  "source": "central",
  "override_enabled": false
}
```

## 5. 적용 전 검증
- 스키마 검증(필수 항목 존재 여부)
- 값 범위 검사(IP/Port/Interval)
- 적용 실패 시 이전 버전 롤백

## 6. 충돌 처리
- 로컬 오버라이드 ON 시:
  - 중앙 업데이트는 보류(경고 로그)
  - 강제 배포(Force)일 경우 로컬 변경을 덮어씀
- 로컬 오버라이드 OFF 시:
  - 중앙 업데이트 즉시 적용

## 7. 오류 처리/백오프
- 네트워크 오류 → 재시도(백오프)
- 인증 실패 → 재시도 중단 + 관리자 알림
- 데이터 손상 → 마지막 정상 버전으로 롤백

## 8. 모니터링 포인트
- 마지막 동기화 시각
- 최신 버전 번호
- 실패 횟수 및 원인
- 현재 적용 중인 설정 소스(central/local)

## 9. 최소 구현 체크리스트
- 주기적 pull 스케줄러
- 설정 변경 시 safe-apply
- ack 전송
- 백오프 로직
- 로컬 메타 기록
