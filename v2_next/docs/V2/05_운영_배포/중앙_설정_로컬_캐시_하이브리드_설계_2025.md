# 중앙 설정 + 로컬 캐시 하이브리드 설계 (현장 PC 1대, 원격 관리)

## 1. 목표
- 원격에서 안전하게 설정을 변경한다.
- 네트워크 장애 시에도 현장 PC가 정상 동작한다.
- 변경 이력/권한/롤백을 체계적으로 관리한다.

## 2. 핵심 개념
- **중앙 설정 서비스 + DB + RBAC**: 권한 통제/감사/버전 관리의 기준점.
- **로컬 캐시/오버라이드**: 오프라인 대응 및 현장 예외 처리.
- 두 구조는 대체가 아니라 **보완 관계**이며 함께 사용해야 안정적이다.

## 3. 구성 요소
1) 중앙 설정 서비스
- 설정 API, 관리 UI, 롤백/버전 관리
- 사용자 권한(RBAC) 및 감사 로그

2) 중앙 DB
- 설정 스냅샷, 변경 이력, 디바이스 상태

3) 현장 PC(에이전트)
- 로컬 캐시(config.ini + meta.json)
- 중앙 설정 동기화(주기적 pull + 필요 시 push)
- 적용 결과 보고(ack)

## 4. 데이터 우선순위(추천)
1) 환경 변수(배포 고정값)
2) 로컬 오버라이드(운영 예외)
3) 중앙 설정(기본값)
4) 코드 기본값

정책: 중앙 설정이 기본값이며, 로컬 오버라이드는 **명시적 허용 모드**에서만 활성화.

## 5. 동기화 흐름
1) 중앙 서비스에서 설정 변경
2) 현장 PC 에이전트가 주기적으로 pull
3) 새 버전 감지 → 로컬 캐시 업데이트
4) 적용 결과 ack 전송
5) 실패 시 롤백 또는 이전 버전 유지

## 6. 충돌/오버라이드 정책
- 로컬에서 변경된 항목은 “오버라이드 모드”일 때만 유지.
- 중앙 업데이트가 강제(Force)일 경우 로컬 오버라이드를 덮어씀.
- 모든 변경은 버전/작성자/시간을 기록.

## 7. 보안 설계
- 디바이스 단위 API 키 발급
- TLS 필수
- RBAC: 관리자/운영자/뷰어 역할 분리
- 모든 변경 사항은 감사 로그 기록

## 8. 장애 대응
- 중앙 연결 불가 시 로컬 캐시 사용
- 마지막 정상 버전(Last Known Good) 유지
- 일정 시간 이상 동기화 실패 시 UI 경고 표시

## 9. 현 시스템 적용 단계(권장 순서)
1) 로컬 캐시 메타 파일 추가  
   - `config_meta.json`에 `version`, `last_sync`, `source` 기록
2) 중앙 설정 API 도입  
   - GET/POST 기준으로 버전 관리 추가
3) 동기화 에이전트 추가  
   - 30~60초 주기 pull, 실패 시 백오프
4) 로컬 오버라이드 모드 도입  
   - UI에서 명시적으로 ON/OFF
5) 감사 로그와 변경 이력 UI 추가

## 10. 최소 운영 모델(현장 PC 1대 기준)
- 중앙 서버 1대 + 단일 디바이스 등록
- 현장 PC는 `config.ini`를 로컬 캐시로 사용
- 원격 변경은 중앙에서 수행, 현장은 자동 동기화
- 긴급 현장 변경 시 로컬 오버라이드 모드로 임시 적용

## 11. 기대 효과
- 원격 관리 안정성 확보
- 네트워크 장애에도 현장 운영 지속
- 변경 이력/권한 관리로 운영 리스크 감소
