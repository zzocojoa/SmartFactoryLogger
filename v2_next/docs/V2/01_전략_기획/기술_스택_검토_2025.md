# 2025년 상용 소프트웨어 기술 스택 검토 리포트

`작성일: 2025-12-31` `작성자: Senior Technical Consultant`
`주제: Python Tkinter의 상용 배포 적합성 및 차세대 전환 전략`

## 1. 질문의 핵심과 답변

**Q: "2025년 상용 배포 제품에 Tkinter를 사용하는 것이 일반적인가?"**

**A: "아니요, 일반적이지 않습니다."** 2025년 시점의 상용 소프트웨어, 특히
**스마트 팩토리/대시보드 분야**에서 Tkinter는 주류 스택이 아닙니다. Tkinter는
**'내부 도구(Internal Tool)'**나 **'프로토타입'**을 빠르게 만들 때는 훌륭하지만,
화려한 UI/UX와 유연성(레이아웃 조절 등)이 필요한 **'상용 제품(Product)'**으로는
한계가 명확합니다.

## 3. 왜 우리는 지금까지 Tkinter를 썼나?

지금까지의 선택이 잘못된 것은 아닙니다. 초기 개발 단계에서는 다음 이유들로 인해
Tkinter가 **최적의 선택**이었습니다.

1. **빠른 개발 속도 (Time-to-Market)**: Python 백엔드 로직과 UI를 한 번에 짤 수
   있어 개발 속도가 매우 빠릅니다.
2. **단일 실행 파일 (.exe)**: 웹 서버 구축 없이, 파일 하나만 던져주면 공장
   PC에서 바로 도는 배포 편의성이 강력합니다.
3. **리소스 절약**: 복잡한 웹 브라우저 엔진 없이 가볍게 돌아갑니다.

**즉, '기능 검증(MVP)' 단계에서는 100점짜리 선택이었습니다.**

## 4. 제언: 향후 전략 (Roadmap)

사용자께서 **"레이아웃 자유 조절"**과 **"상용 수준의 퀄리티"**를 요구하기
시작했다면, 이제 **기술 스택의 전환(Pivot)**을 고민해야 할 시점입니다.

### 4.1. 단기 전략 (Phase 1 Complete - 이번 프로젝트)

- **목표**: 현재의 Tkinter 기반을 잘 마무리하여 **'안정적인 버전 1.0'** 출시.
- **대응**: 무리하게 드래그 앤 드롭을 구현하기보단, **'프리셋(Preset)'
  제공**으로 타협하여 완성도에 집중.
- **이유**: 당장 스택을 엎으면 출시가 수개월 지연됨.

### 4.2. 중장기 전략 (Phase 2 - 차세대 버전)

- **목표**: **웹 기술(웹 기술) 도입**으로 전환.
- **구조**:
  - **백엔드**: Python (FastAPI/Flask) - 데이터 수집 및 로직 담당. (현재 코드
    재사용 가능)
  - **프론트엔드**: React.js 또는 Vue.js - 미려한 UI 및 자유로운 레이아웃 담당.
  - **Wrapper**: Electron 또는 Tauri - 웹 화면을 데스크톱 앱처럼 감싸서 배포.
- **기대 효과**:
  - 사용자가 원하는 **'위젯 크기 조절/배치 자유'** 기능 기본 제공 (ex:
    `react-grid-layout`).
  - 공장장실이나 스마트폰에서도 대시보드 확인 가능 (원격 모니터링).
  - 압도적인 디자인 퀄리티 상승.


### 4.3. 2025 실무 보완 포인트
- 운영/현장 조건을 스택 결정의 1차 기준으로 명시: 오프라인, 24/7, 복구 시간, 입력 장치, 멀티 모니터.
- V1은 안정성과 배포 단순성, V2는 확장성과 유지보수성에 목표를 분리해 일정 리스크를 낮춘다.
- 전환 비용 최소화를 위해 수집/저장/가공 로직을 서비스/API로 분리하고 UI는 계약 기반으로 교체한다.
- 현장 장치 접근(PLC/카메라/시리얼)은 백엔드 서비스가 담당하고, UI는 API로만 접근하도록 역할을 분리한다.
- 보안/업데이트: 코드 서명, 오프라인 업데이트 절차, 오류/성능 로그 수집 및 현장 통보 체계를 포함한다.
- 기업 보안 정책(GPO/백신/SmartScreen)을 고려한 실행 예외 등록과 사전 검증 체크리스트를 포함한다.
- 웹 기술 선택 시 Electron vs Tauri 비교 기준(메모리, 업데이트, 보안, 운영자 PC 사양)을 문서화한다.
