# 설정 핫 리로드 검증 시나리오

## 1. 목적
- 재시작 없이 설정 변경이 실제 동작에 반영되는지 검증한다.
- Extruder/LS/Spot/Logger의 변경 경로를 분리해 증빙을 남긴다.

## 2. 실행 환경
- 개발 PC(MOCK) + 현장 PC(REAL) 병행 검증
- 공통 환경 변수: `SFL_CONFIG_PATH`, `V2_MODE`

## 3. 검증 범위
- Extruder: IP/Port 변경 핫 리로드
- LS PLC: IP/Port + 대상 매핑 변경 핫 리로드
- SPOT: URL/Refresh/위젯 파라미터 핫 리로드
- Logger: logpath/autosave/rotation/header 핫 리로드

## 4. 공통 준비 절차
1) 백엔드 실행(현장: REAL / 개발: MOCK)
2) `/health` 확인(스레드/드라이버 상태)
3) `system.log` 및 `comm_metrics.log` 위치 확인

## 5. 시나리오별 검증 절차

### 5.1 Extruder 핫 리로드
- 변경 항목: `EXTRUDER.ip`, `EXTRUDER.port`
- 절차:
  1) `config.ini`에서 IP/Port 변경
  2) `system.log`에 applied 기록 확인
  3) `comm_metrics.log`에서 `EX_DISCONNECTED` → `EX_RECOVERED` 이벤트 확인
  4) `/health`의 comm metrics에서 연결 상태 복구 확인
- 기대 결과:
  - 소켓 재연결이 수행되고 복구 시간/횟수가 기록됨
- 실행 상태: 현장 테스트 필요(REAL)

### 5.2 LS PLC 핫 리로드
- 변경 항목: `LS_PLC.ip`, `LS_PLC.port`, `LS_PLC_TARGETS`
- 절차:
  1) `config.ini`에서 IP/Port 변경
  2) `system.log` applied 기록 확인
  3) `comm_metrics.log`에서 `LS_DISCONNECTED` → `LS_RECOVERED` 이벤트 확인
  4) 타깃 매핑 변경 후 다음 read 사이클의 데이터 반영 확인
- 기대 결과:
  - 재연결 및 매핑 적용이 정상 수행됨
- 실행 상태: 현장 테스트 필요(REAL)

### 5.3 SPOT 핫 리로드
- 변경 항목: `SPOT.ip`, `SPOT.refreshinterval`, `SPOT.imageurl`, 위젯/크로스헤어 파라미터
- 절차:
  1) `refreshinterval` 변경 후 `/api/spot/config` 값 확인
  2) `imageurl`을 임시 오류 URL로 변경 → 오류 상태 표시 확인
  3) 정상 URL로 복구 → 상태 정상화 확인
- 기대 결과:
  - 백엔드는 즉시 적용, 프론트는 재조회 후 반영
- 실행 상태: 개발 PC 확인 가능(모의/현장 공통)

### 5.4 Logger 핫 리로드
- 변경 항목: `SETTINGS.logpath`, `SETTINGS.autosave`, `LOGGING.rotationmode`,
  `LOGGING.cycleidletime`, `LOGGING.cyclethresholdpress`, `HEADERS.csv`
- 절차:
  1) `logpath` 변경 → 새 경로에 로그/CSV 생성 확인
  2) `autosave`/`rotationmode` 변경 → 즉시 반영 로그 확인
  3) `HEADERS.csv` 변경 → 다음 파일 생성 시 헤더 반영 확인
- 기대 결과:
  - 파일 핸들 재오픈 + 적용 로그 기록
- 실행 상태: 개발 PC 확인 가능(모의/현장 공통)

## 6. 실행 결과 요약(현재)
- Extruder: 미수행(현장 테스트 필요)
- LS PLC: 미수행(현장 테스트 필요)
- SPOT: 미수행(개발 PC 가능)
- Logger: 미수행(개발 PC 가능)

- `comm_metrics.log`의 DISCONNECTED/RECOVERED 이벤트
- 관측성 내보내기(`observability_snapshot_*.json`)
