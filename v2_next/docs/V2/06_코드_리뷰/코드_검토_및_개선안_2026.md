# V2.0 코드 검토 및 개선안 (2026.01.01)

## 1. 종합 평가
- 서비스 레이어 구조가 명확하며, PLC/Spot/Logger 분리가 잘 되어 있다.
- 핫 리로드 도입으로 운영 편의성은 크게 향상되었으나, 적용 범위/예외에 대한
  경계 처리가 필요하다.

## 2. 주요 개선 포인트

### 2.1 백엔드

1) 핫 리로드 적용 결과 표준화
- 적용 성공/보류(pending) 기준을 명확히 정의해야 한다.
- 설정 저장 API 응답에 applied/pending이 포함되므로 UI와 일관성 유지 필요.

2) 설정 변경 watcher 안정성
- config.ini가 빈 파일로 잠깐 바뀌는 경우에도 안전하게 복구되도록
  재시도/지연 로직 검토 필요.

3) 임계값(thresholds) 백엔드 처리
- 현재 UI에서는 적용되지만, 백엔드의 상태 판단 로직에는 미반영이다.
- 알림/이상 감지 로직이 필요한 경우 서비스 레이어에 별도 반영 필요.

4) 설정 비밀번호 적용 결과
- 실제로는 즉시 반영되지만 pending으로 표시될 수 있다.
- 적용 결과 분류 로직 개선 필요.

5) 재연결 시 데이터 공백 대비
- PLC IP/Port 변경 후 재연결 과정에서 데이터 공백이 발생한다.
- 재연결 상태를 UI에 노출하거나 백오프 상태 로그를 강화할 필요가 있다.

### 2.2 프론트엔드

1) 외부 config.ini 변경 반영
- watcher가 적용하더라도 UI는 즉시 갱신되지 않는다.
- 주기적 재조회/갱신 버튼/웹소켓 알림 등 보완 필요.

2) 적용 결과 가시성
- applied/pending 요약은 표시되나 세부 항목 리스트는 없음.
- 운영 담당자를 위해 상세 항목 보기 제공 검토 필요.

## 3. 리스크 및 대응
- hot reload 범위를 확대할수록 “부분 적용” 상태가 늘어날 수 있음.
- 운영 배포 시에는 “즉시 적용 항목 vs 재시작 필요 항목”을 명확히 안내해야 함.
- watcher 자동 적용은 편의성이 높으나, 잘못된 config 변경이 즉시 반영될 리스크 존재.

## 4. 권장 우선순위
1) thresholds 백엔드 로직 반영 여부 결정
2) 설정 비밀번호 적용 결과 분류 개선
3) watcher 안정성/로그 강화
4) UI 재조회 정책 추가

## 5. 결론
핫 리로드 구조는 운영 관점에서 큰 장점이 있으나, 적용 범위와 예외 처리를 명확히
정리해야 한다. 설정 변경이 즉시 반영되는 만큼, 운영 문서와 UI 가시성 개선이 필수다.
