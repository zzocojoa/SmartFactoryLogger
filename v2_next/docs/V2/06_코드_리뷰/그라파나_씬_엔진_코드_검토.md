# 그라파나 씬 엔진 코드 검토 리포트

## 1. 검토 범위
- `v2_next/frontend/src/App.tsx`
- `v2_next/frontend/src/scenes/DashboardScene.ts`
- `v2_next/frontend/src/scenes/ScenesRuntime.tsx`

## 2. 주요 발견 사항 (심각도 순)
- 높음: 레이아웃 저장/복원이 인덱스 기반이라 위젯 순서 변경/추가 시 다른 카드에 배치가 적용될 수 있음. 저장 데이터에 위젯 식별자가 없고, 로딩 시 `defaultChildren`의 인덱스에 매핑함. `v2_next/frontend/src/App.tsx:151`, `v2_next/frontend/src/App.tsx:157`, `v2_next/frontend/src/scenes/DashboardScene.ts:38`
- 중간: 레이아웃 저장 시 `grid.state`를 `any` 캐스팅으로 직접 접근. 라이브러리 내부 구조 변경 시 깨질 위험이 큼. `v2_next/frontend/src/App.tsx:153`
- 중간: `initScenesRuntime()`가 모듈 최상단에서 즉시 실행됨. HMR/테스트에서 중복 초기화 가능성이 있어 런타임 오염 위험. `v2_next/frontend/src/App.tsx:11`
- 중간: 공지/헤더 한글 문자열이 깨져 있어 운영 화면 품질 저하. 인코딩 문제가 재발 가능. `v2_next/frontend/src/App.tsx:81`, `v2_next/frontend/src/App.tsx:171`
- 낮음: `DataContext`가 `any`로 선언되어 타입 안전성이 낮음. 런타임 오류를 조기에 잡기 어렵다. `v2_next/frontend/src/App.tsx:194`
- 낮음: `sceneRenderers`가 선언만 되어 있고 사용되지 않아 유지보수 혼선 유발. `v2_next/frontend/src/App.tsx:116`
- 낮음: 레이아웃 저장 시 `alert` 사용으로 UI가 블로킹됨. 현장용 화면에서는 비동기 알림이 적합. `v2_next/frontend/src/App.tsx:163`
- 낮음: `SceneFlexLayout`, `SceneFlexItem` 등 사용하지 않는 import가 남아 있음. `v2_next/frontend/src/scenes/DashboardScene.ts:5`
- 낮음: `mockBackendSrv.datasourceRequest`가 항상 빈 배열을 반환. 향후 씬 쿼리 기능을 붙이면 원인 파악이 어려운 “무응답 정상” 상태가 될 수 있음. `v2_next/frontend/src/scenes/ScenesRuntime.tsx:26`

## 3. 개선 제안 (우선순위)
1) 레이아웃 저장 형식을 `id` 기반으로 변경: 각 카드에 고정 `key`(예: kpi, spot, temps, molds, camera, notice)를 부여하고, 저장 시 `{id: {x,y,w,h}}` 형태로 기록한 뒤 로딩 시 id 매핑으로 복원.
2) 레이아웃 저장 로직을 내부 상태 접근 대신 공식 API/상태 구독 방식으로 전환: 라이브러리 업데이트에 대한 파괴적 영향 최소화.
3) `initScenesRuntime`는 앱 마운트 시 1회만 실행되도록 가드(예: `useRef` 플래그) 처리.
4) 공지/헤더 문자열의 인코딩을 UTF-8로 정규화하고, 텍스트 리소스를 별도 상수/파일로 분리해 재발 방지.
5) `DataContext` 타입 정의 추가 + 미사용 코드/임포트 정리 + 저장 시 비블로킹 알림으로 교체.
6) 씬 기반 데이터 소스를 실제로 사용할 계획이 있다면, `datasourceRequest`를 실제 백엔드 호출로 연동하거나 최소한 미구현 오류를 명시하여 디버깅성을 확보.

## 4. 확인 필요 / 질문
- 씬(Scenes) 내부 데이터 소스/쿼리 기능을 실제로 사용할 계획이 있는지? 있다면 `ScenesRuntime`의 mock 전략을 언제 실제 연결로 전환할지 결정이 필요.
- 레이아웃 저장 파일(`layout.json`)의 버전/마이그레이션 전략이 필요함. 추후 위젯이 추가/삭제될 때의 초기화 기준을 정의할지 여부 확인. 로컬 스토리지는 1회 마이그레이션 용도.

## 5. 결론
현재 구조는 “씬 기반 레이아웃 + React 위젯 렌더링” 흐름은 정상 동작 가능하지만, 레이아웃 저장/복원 방식의 취약성과 런타임 초기화/인코딩 문제는 현장 운영에서 재현될 확률이 높다. 위 1~4번을 우선 적용하면 안정성과 유지보수성이 크게 향상된다.
