# V2.0 코드 검토 및 개선 제안서 (2026.01.01)

## 1. 종합 평가 (Overall Assessment)

**"체계적인 레이어 분리로 확장성 확보, 단 운영 환경을 위한 견고함(Robustness)
보강 필요"**

작성된 `v2_next` 코드는 **Service Layer Pattern**을 충실히 따르고 있어,
V1(Legacy)의 스파게티 코드 문제를 성공적으로 해결했습니다. 특히 **FastAPI +
Background Thread** 구조는 비동기 웹 프레임워크와 Blocking I/O(PLC 통신)를
조화시키는 모범적인 패턴입니다.

다만, 현장 상용 가동(24/7)을 위해서는 **로깅 체계, 정밀 타이밍, 예외 처리의
세분화** 측면에서 보완이 필요합니다.

---

## 2. 주요 장점 (Key Strengths)

1. **아키텍처 분리 (Separation of Concerns)**:
   - `main.py`는 HTTP 요청 처리만, `plc_service.py`는 주기적 수집만,
     `real_driver.py`는 하드웨어 프로토콜만 담당하도록 역할이 명확합니다.
2. **설정 유연성 (Configuration)**:
   - `config.py`에서 파일 설정과 환경 변수(`os.getenv`)를 모두 지원하여,
     개발(Mock)과 현장(Real) 간 전환이 `ENV` 변수 하나로 가능합니다.
3. **스레드 안전성 (Thread Safety)**:
   - `threading.Lock`을 사용하여 Background Thread와 API Request Thread 간의
     데이터 경합(Race Condition)을 방지했습니다.

## 3. 개선 권고 사항 (Recommendations)

### 3.1. 백엔드 (Backend)

#### (1) `print()` 대신 `logging` 모듈 사용 (Critical)

현재 `real_driver.py`와 `plc_service.py`에서 에러 발생 시 `print()`문을 사용하고
있습니다. 이는 콘솔에서만 확인 가능하며, 현장 문제 발생 시 원인 파악을 어렵게
합니다.

- **제안**: Python 내장 `logging` 모듈을 도입하여 파일(`logs/system.log`)로
  기록하고, Log Rotation(날짜별/용량별 분할)을 적용하세요.

#### (2) 수집 주기 정밀도 향상 (`plc_service.py`)

현재 로직은 `read_data()` 수행 시간 + `sleep(0.2)` 입니다. 통신이 0.1초 걸리면
실제 주기는 0.3초가 되어 데이터가 밀립니다.

- **As-Is**:
  ```python
  # 통신 시간(Variable) + 0.2s
  time.sleep(config.INTERVAL_SEC)
  ```
- **To-Be (Drift Correction)**:
  ```python
  next_wake = time.time() + config.INTERVAL_SEC
  # ... read_data ...
  sleep_time = max(0, next_wake - time.time())
  time.sleep(sleep_time)
  ```

#### (3) SPOT 센서 타임아웃 격리 (`real_driver.py`)

`read_data` 함수 내에서 PLC(소켓)와 SPOT(HTTP)을 순차 실행합니다. 만약 SPOT
네트워크 오류로 0.2초 타임아웃이 발생하면, PLC 데이터 수집도 그만큼 지연됩니다.

- **제안**: `SPOT` 수집은 별도 스레드로 분리하거나, 실패 시 즉시 넘어가도록
  `Requests`의 `timeout`을 매우 짧게(0.05s~0.1s) 잡거나 Non-blocking IO로
  전환하는 것을 고려해야 합니다.

### 3.2. 프론트엔드 (Frontend)

#### (1) `any` 타입 캐스팅 지양 (`App.tsx`)

`const ResponsiveGridLayout = Responsive as any;`는 라이브러리 호환성을 위한
임시 방편입니다.

- **제안**: `react-grid-layout`의 `@types` 패키지 버전을 맞추거나, `d.ts` 정의
  파일을 추가하여 타입 안전성을 확보하는 것이 좋습니다.

#### (2) 폴링 주기 동기화

백엔드는 0.2초(5Hz)로 수집하지만, 프론트엔드는 0.5초(2Hz)로 `setInterval`이
설정되어 있습니다. 빠른 변화를 감지해야 한다면 프론트 주기를 200ms로 낮추는 것을
검토하세요.
