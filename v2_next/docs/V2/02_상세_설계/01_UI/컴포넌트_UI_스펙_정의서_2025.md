# 컴포넌트 UI 스펙 정의서 (압출 공정 대시보드, 2025)

## 1. 목적과 범위

- 실시간 대시보드 카드 컴포넌트의 데이터, 레이아웃, 상태 표시 규칙을 표준화한다.
- 범위는 현재 구현 중인 카드형 UI 전체(공정 KPI, SPOT, 온도군, 몰드, 환경,
  카메라, 알림)에 한정한다.

## 2. 공통 UI 토큰

- **타이포**: 숫자 `clamp(24px, 4vw, 40px)`, 보조 텍스트 12~14px, `tabular-nums`
  적용.
- **간격**: 카드 내부 패딩 12~16px, 카드 간 간격 12~16px.
- **색상**: 정상(청록/녹색), 주의(황색), 위험(적색), 비활성(회색), 배경은 저채도
  다크.
- **단위**: 숫자 오른쪽 고정 위치, 줄바꿈 금지.
- **결측**: 값이 없으면 `--` 표시 + 회색 톤 + 마지막 갱신 시간 표시.

## 3. 공통 상태 규칙

- **정상**: 기본 색상, 상태 라벨 없음 또는 “정상”.
- **주의**: 노란색 배지 + 텍스트 “주의”, 아이콘 병행.
- **위험**: 빨간색 배지 + 텍스트 “위험”, 아이콘 병행.
- **연결 끊김**: 카드 전체 톤 다운 + “연결 끊김” 라벨.
- **지연**: “지연됨(갱신 HH:MM:SS)” 표시.

## 4. 컴포넌트별 스펙

### 4.1 공정 KPI 카드 (KpiComponent)

- **ID**: `kpi`
- **데이터**: `Speed`, `Press`, `Count`, `EndPos`
- **기본 크기**: 6x8
- **최소 크기**: 5x6
- **표현 규칙**: 속도/압력은 가장 큰 수치 + **얇은 가로 Bar**, 카운트/종료
  위치는 보조 영역.
- **포맷**: Speed/Press 1자리, Count 정수, EndPos 1자리.
- **상태 규칙**: Speed 또는 Press가 0 지속 시 “정지” 배지 표시.
- **속도 라벨**: `0: 대기`, `0<~<2 매우저속`, `2~<4 저속`, `4~<6 보통`,
  `6~<8 빠름`, `>=8 매우빠름`
- **압력 라벨**: `0: 대기`, `0<~<126 낮음`, `126~<180 보통`, `>=180 높음`
- **매핑 원칙**: 속도/압력 라벨은 Normal/Warning/Danger로 치환하지 않음.
- **라벨 색상**: 매우빠름=적색, 빠름=황색, 보통=청록, 저속=청색,
  매우저속=회청색.

### 4.2 SPOT Temperature 카드 (SpotComponent)

- **ID**: `spot`
- **데이터**: `Spot`
- **기본 크기**: 10x4
- **최소 크기**: 8x4
- **표현 규칙**: 중앙 대형 수치 + **Arc Gauge** 배경 + 하단 상태 라벨.
- **임계치**: 위험 기준 500°C, 정상/주의/위험 라벨 병행.
- **포맷**: 1자리 + 단위 `°C` 고정.
- **상태 규칙**: `0: 대기`, `0<~<480 저온`, `480~<540 보통`, `540~<580 고온`,
  `>=580 경고`.
- **경고 강조**: 580°C 이상 시 **정적 테두리 + 경고 아이콘** 적용.

### 4.3 Secondary Temps 카드 (TempsComponent)

- **ID**: `temps`
- **데이터**: `Temp_F`, `Temp_B`, `Billet_Temp`, `Billet_Length`
- **기본 크기**: 10x4
- **최소 크기**: 8x4
- **표현 규칙**: 2x2 서브 그리드, 온도는 상단, 길이는 하단.
- **포맷**: 온도 1자리, 길이 0~1자리.

### 4.4 Mold Zones 카드 (MoldsComponent)

- **ID**: `molds`
- **데이터**: `Mold1`~`Mold6`
- **기본 크기**: 8x8
- **최소 크기**: 7x6
- **표현 규칙**: 3x2 그리드, 몰드 번호 라벨 고정.
- **상태 규칙**: 이상값 몰드만 강조, 다른 몰드는 기본 유지.

### 4.5 환경 카드 (At Temp / At Pre)

- **ID**: `env`
- **데이터**: `At_Temp`, `At_Pre`
- **기본 크기**: 4x3
- **최소 크기**: 4x2
- **표현 규칙**: 소형 카드 2개 또는 2열 배치, 상태 라벨 병행.
- **포맷**: 온도 1자리, 압력 1자리.

### 4.6 SPOT Camera 카드 (CameraComponent)

- **ID**: `camera`
- **데이터**: `spotImageUrl`, `spotConfig`
- **기본 크기**: 12x10
- **최소 크기**: 8x6
- **표현 규칙**: 영상 비율 고정, 중앙 로딩/연결 오버레이(텍스트 + 스피너).
- **인터랙션**: 포커스 +/- 버튼, 입력 중 비활성화 표시.

### 4.7 알림 카드 (Operator Check)

- **ID**: `notice`
- **데이터**: 운영 메시지 텍스트
- **기본 크기**: 12x10
- **최소 크기**: 8x6
- **표현 규칙**: 아이콘 + 제목 + 행동 지시 문장.
- **상태 규칙**: 경고 시 테두리/배경 강조, 비활성 시 톤 다운.

## 5. 리사이즈 규칙

- 최소 크기 이하에서는 보조 텍스트 숨김.
- 크기 확대 시 수치만 확대하고 카드 레이아웃은 유지.
- 텍스트 잘림 방지를 위해 단위와 숫자를 동일 라인에 고정.

## 6. 구현 체크 포인트

- 드래그/리사이즈는 편집 모드에서만 활성화.
- 숫자 포맷은 전 카드 동일 규칙 적용.
- 상태 색상은 공통 토큰을 사용.

| Mold Zones                   | 기본 톤                             | 해당 셀만 경고 테두리/텍스트          | 해당 셀만 굵은 테두리/아이콘                          | v1은 warning 색상만 명시, Danger 기준은 설정 필요                                       |

## 8. KPI/카메라 임계값 기준 제안 (v2 정책)

### 8.1 KPI 임계값 (속도/압력/카운트/종료 위치)

- **원칙**: 단일 샘플에 반응하지 않고, 0.2초 주기 기준 최소 10~25샘플(2~5초)
  지속 시 상태 전환.
- **Speed**
  - Warning: `speed < min_speed` 또는 `speed > max_speed`가 2~3초 지속.
  - Danger: `speed == 0`이면서 `press >= press_running_threshold`가 2~3초
    지속(정지/막힘).
  - 기준값은 라인별 설정값 우선, 없으면 최근 5분 중앙값 대비 ±30%를 Warning,
    ±50%를 Danger로 적용.
- **Press**
  - Warning: `press > max_press` 또는 `press < min_press`가 2~3초 지속.
  - Danger: `press >= press_hard_limit` 또는 `press == 0`이면서
    `speed > min_speed`가 2~3초 지속.
- **Count**
  - Warning: `speed > min_speed` && `press > press_running_threshold`인데도
    카운트 증가 없음이 10초 이상 지속.
  - Danger: 카운트 역행/리셋(단, 정상 사이클 리셋 구간은 제외).
- **End Position**
  - Warning: `endpos`가 허용 범위를 벗어난 상태가 2~3초 지속.
  - Danger: `endpos` 급격한 역방향 점프(허용 오차 초과).

### 8.2 SPOT Camera 임계값 (연결/지연)

- **Warning**
  - 새 프레임 미도착 시간이 `2 x refresh_interval` 초과.
  - 연속 오류(HTTP/IO) 3회 이상.
- **Danger**
  - 새 프레임 미도착 시간이 `5 x refresh_interval` 초과.
  - 연속 오류 10회 이상.
- **표현 규칙**
  - Warning: “지연됨” + 아이콘 + 상태색(Warning).
  - Danger: “연결 끊김” + 아이콘 + 상태색(Danger).

## 9. 결론

컴포넌트 스펙을 고정하면 UI 일관성과 현장 대응성이 동시에 개선된다. 특히 상태
표현과 숫자 포맷의 통일은 운영 오류를 줄이는 핵심 요소다.
