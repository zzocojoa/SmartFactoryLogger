# 설정 UI 레이아웃 저장 서버/파일 설계 (2025)

## 1. 목적
- 편집 모드에서 변경된 레이아웃을 **브라우저 로컬 스토리지** 대신 **백엔드 서버가 파일로 저장**하도록 전환한다.
- 이식 PC에서도 동일한 저장 위치/복구 정책을 유지한다.

## 2. 저장 범위
- 저장 대상: 카드 레이아웃 좌표(`x,y,w,h`)와 컬럼 정보.
- 저장 단위: 카드 고정 ID 기준 맵(`{id: {x,y,w,h}}`).
- 저장 슬롯: **최대 3개**까지 보관하며, 사용자가 **이름을 지정**한다.

## 3. 저장 위치/경로 규칙
- 기준 경로: `SFL_CONFIG_PATH` 환경변수가 있으면 해당 파일의 **디렉터리**를 사용.
- 환경변수가 없으면 기본 경로:
  `C:\Users\<user>\AppData\Roaming\SmartFactoryLogger\`
- 레이아웃 파일:
  - 본 파일: `layout.json`
  - 백업 파일: `layout.backup.json`
  - 임시 파일: `layout.tmp` (원자적 저장용)

## 4. 데이터 스키마 (파일)
```json
{
  "version": "v2",
  "updated_at": "2026-01-02T10:15:00+09:00",
  "active_id": "slot1",
  "slots": [
    {
      "id": "slot1",
      "name": "기본",
      "updated_at": "2026-01-02T10:15:00+09:00",
      "cols": "60",
      "layout": {
        "kpi": { "x": 0, "y": 0, "width": 15, "height": 18 },
        "temps": { "x": 15, "y": 4, "width": 25, "height": 4 },
        "molds": { "x": 40, "y": 0, "width": 20, "height": 8 }
      }
    }
  ]
}
```

## 5. 서버 API 설계
- `GET /api/layout`
  - 응답: 활성 레이아웃(JSON). 없으면 404.
- `GET /api/layouts`
  - 응답: 저장된 레이아웃 목록(최대 3개) + 활성 ID.
- `POST /api/layouts`
  - 요청: `{ name, layout, cols, version, slot_id? }`
  - 동작: 레이아웃 슬롯 저장(동일 이름이면 업데이트), 3개 초과 시 400.
- `POST /api/layouts/restore`
  - 요청: `{ slot_id }`
  - 동작: 선택한 슬롯을 활성 레이아웃으로 변경.
- `POST /api/layouts/delete`
  - 요청: `{ slot_id }`
  - 동작: 슬롯 삭제 + 활성 슬롯 재설정(남은 첫 슬롯 또는 null).
- `POST /api/layout/restore`
  - 동작: `layout.backup.json`을 `layout.json`으로 복구(비상 복구용).
- `GET /api/layout/meta`
  - 응답: 파일 경로, 마지막 저장 시각, 버전/활성 ID/슬롯 수.

## 6. 로딩/저장 동작
1) 앱 시작 시 `GET /api/layout` 호출.
2) 파일이 없으면 기본 레이아웃 적용.
3) 저장 버튼 클릭 시 이름 입력 → `POST /api/layouts` 호출.
4) 목록에서 복구 대상 선택 → `POST /api/layouts/restore` 호출.
5) 목록에서 삭제 대상 선택 → `POST /api/layouts/delete` 호출.

## 7. 로컬 스토리지 마이그레이션(1회)
- 서버에 레이아웃 파일이 없고, 로컬 스토리지에 기존 레이아웃이 있으면:
  1) 로컬 스토리지 값을 `이전 레이아웃` 이름으로 업로드
  2) 서버 저장 성공 시 로컬 스토리지 키 제거

## 8. 안정성/예외 처리
- 저장은 **임시 파일 → 원본 교체** 방식으로 원자적 처리.
- 권한 오류/읽기 전용 등 실패 시:
  - UI에 “저장 실패(권한/경로 확인)” 안내 표시.
  - 기존 레이아웃을 유지하고 데이터 손실 방지.

## 9. 운영 고려사항
- 레이아웃 파일은 config.ini와 동일한 경로를 사용하여 **이식 PC에서 관리 일원화**.
- 원격 점검 시 `layout.json`을 함께 수집하면 레이아웃 분석 가능.

## 10. 검증 항목
- 저장/복구 정상 동작 (슬롯 저장/선택 복구).
- 슬롯 최대 3개 제한 동작.
- 슬롯 삭제 후 목록/활성 ID 갱신 동작.
- 서버 재시작 후 레이아웃 유지.
- 환경변수 경로 변경 시 저장 경로 변경 반영.
- 파일 읽기 전용 상황에서 적절한 오류 표시.
