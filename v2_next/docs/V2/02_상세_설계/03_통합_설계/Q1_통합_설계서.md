# V2.0 Q1 Plan: Real Data Integration Design

`작성일: 2025-12-31`
`목표: V2 백엔드(FastAPI)와 실제 공장 설비(PLC/Sensor) 간의 연동 아키텍처 설계`

## 1. 배경 및 목표

현재 V2 Prototype은 내부 시뮬레이션 로직(Random)으로 돌아가고 있습니다. 2026
Q1의 핵심 목표는 이를 **실제 데이터(Real Data)**로 교체하여 상용 수준의 데이터
파이프라인을 구축하는 것입니다.

## 2. 기존(V1) 분석 vs 신규(V2) 설계

### 2.1. V1 Legacy 구조 분석

- **위치**: `v1_legacy/src/modules/`
- **방식**: Tkinter GUI 루프(`after`) 내에서 직접 폴링하거나, `threading`(배경
  스레드)이 혼재됨.
- **주요 모듈**:
  - `melsec_driver.py` (또는 유사 파일): Extruder 메인 PLC 통신.
  - `ls_driver.py`: 온도 컨트롤러 통신.
  - `spot_sensor.py`: 적외선 온도계 HTTP 요청.

### 2.2. V2 Next 아키텍처 (계층형 아키텍처)

FastAPI의 장점을 살리기 위해 **서비스 레이어** 패턴을 도입하여 통신 로직을
Controller(API)와 분리합니다.

```mermaid
graph TD
    PLC[Real PLC/Sensors] -->|Modbus/TCP| Driver[Driver Layer]
    Driver -->|Dict/Model| Service[PLC Service (Background Thread)]
    Service --Store--> State[Global State (Memory)]
    Client[React Frontend] <--Polling 0.5s--> API[FastAPI Endpoints]
    API --Read--> State
```

## 3. 구현 상세 설계

### 3.1. Directory Structure (백엔드)

```text
v2_next/backend/
├── main.py                # App Entrypoint
├── config.py              # Configuration (IP, Port)
├── models/
│   └── data_model.py      # Pydantic Schemas (FactoryData)
└── services/
    ├── __init__.py
    ├── base_driver.py     # (New) Abstract Base Class
    ├── mock_driver.py     # (Current) Simulation Logic
    └── real_driver.py     # (New) Ported Logic from V1
```

### 3.2. Data Flow Strategy (Background Task)

FastAPI는 비동기(Async) 처리에 강점이 있지만, **Modbus/TCP** 등 레거시
프로토콜은 Blocking I/O인 경우가 많으므로 별도 **데몬 스레드(Daemon
Thread)**에서 수집 루프를 돌립니다.

1. **Service 시작**: `main.py`의 `@app.on_event("startup")` 시점에 `PLCService`
   스레드 시작.
2. **데이터 수집**: 0.2초(설정 가능)마다 PLC 레지스터 Read.
3. **상태 갱신**: 최신 값을 메모리 상의 `Global State` 객체에 덮어쓰기 (스레드
   안전성 확보를 위해 Lock 사용 고려).
4. **API 응답**: 프론트엔드 요청 시 메모리 상태 즉시 반환 (Zero Latency).

### 3.3. Configurable Driver (설정 기반 드라이버 교체)

- 실행 모드: `V2_MODE=REAL`이면 Real, 그 외는 MOCK.
- 설정 파일 우선순위: `SFL_CONFIG_PATH` 지정 -> 현재/상위 디렉토리 `config.ini`
  탐색.
- 환경 변수 오버라이드: `EXTRUDER_IP/PORT`, `LS_IP/PORT`, `SPOT_IP`, `SPOT_URL`,
  `SPOT_IMAGE_URL`, `SPOT_REFRESH_INTERVAL`, `SPOT_CROSSHAIR_*`,
  `SPOT_ACTUATOR_*`.
- 수집 주기 정책: `INTERVAL_SEC`는 0.2초로 고정(현장 정책 반영).

```python
# Pseudo Code
mode = os.getenv("V2_MODE", "MOCK").upper()
driver = RealPLCDriver() if mode == "REAL" else MockPLCDriver()

config_path = os.getenv("SFL_CONFIG_PATH")
```

## 5. 구현 결과(현황)

### 5.1 반영된 내용(완료)

- V2 백엔드 기본 동작: FastAPI + 서비스 레이어 구조로 API와 수집 로직 분리.
- 실행 모드 전환: `V2_MODE=REAL|MOCK`로 Real/Mock 드라이버 선택.
- 설정 파일 로딩: `SFL_CONFIG_PATH` 우선, 다중 경로 탐색 + 인코딩(UTF-8/CP949)
  대응.
- 환경 변수 오버라이드: EXTRUDER/LS/SPOT IP 및 포트/URL 덮어쓰기 지원.
- `INTERVAL_SEC` 0.2초 고정 적용 및 `PLCService` 주기 동기화.
- 헬스 체크: `/health`에서 스레드 상태/마지막 업데이트/연결 상태 제공.
- 통신 기본 안정화: Melsec `recv_until`, LS `recv_exact`, 타임아웃 처리 적용.
- SPOT 설정 확장: `imageurl`, `refreshinterval`, `crosshair*`, `focusurl`,
  `actuatorip/step`, `widget*` 로딩.
- SPOT 제어 API 추가: `/api/spot/config`, `/api/spot/focus`.
- 프론트 카메라 뷰 구현: 이미지 갱신, 크로스헤어 오버레이, 포커스 버튼 연동.
- 프론트 UI 텍스트 깨짐(인코딩) 수정.
- 정합성 비교 절차 문서화: V1/V2 매핑, 샘플링, 허용 오차, PASS/FAIL 기준 정리.
- 실행/배포 문서 정리: `v2_next/README.md`에 NAS(Z:)→Desktop 복사 최적화와 루트
  실행 규칙 반영.
- 설정 핫 리로드: config.ini 변경 감지 → 변경 diff 계산 → 즉시 적용/보류 분류.
- 로그/저장 핫 리로드: logpath/snapshotpath/autosave/rotation/CSV 헤더 즉시 반영.
- SPOT 핫 리로드: IP/URL/refresh 및 crosshair/위젯/액추에이터 설정 즉시 반영.
- PLC 통신 핫 리로드: Extruder/LS IP·Port 변경 시 소켓 재연결.
- 설정 저장 응답에 적용 결과(applied/pending) 포함.

### 5.2 부분 반영(추가 보완 필요)

- SPOT 포커스 제어: API/버튼 연결은 완료되었으나 실제 액추에이터 동작 검증 필요.

- 통신 검증: 기본 파싱은 적용했으나 블록 병합 최적화/정밀 검증 로직은 미구현.
- 현장 검증: 포트 연결 확인은 완료되었으나 V1 vs V2 데이터 정합성 비교는 미완.

- config.ini 외부 변경 시 프론트 UI 자동 반영은 미구현(재조회 필요).
