## 1. 사전 준비

### 1.1 Chrome DevTools 열기

**방법 1**: 키보드 단축키

- Windows: `F12` 또는 `Ctrl + Shift + I`
- Mac: `Cmd + Option + I`

**방법 2**: 마우스

1. 페이지에서 우클릭
2. "검사" 또는 "Inspect" 클릭

### 1.2 DevTools 화면 이해

```
┌─────────────────────────────────────────────────────────┐
│  Elements | Console | Sources | Network | Performance  │  ← 탭들
├─────────────────────────────────────────────────────────┤
│                                                         │
│                    분석 결과 표시 영역                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.3 분석 환경 설정 (중요!)

정확한 측정을 위해:

1. **시크릿 모드 사용** (Ctrl+Shift+N)
   - 확장 프로그램 영향 제거

2. **캐시 비활성화**
   - DevTools → Network 탭 → "Disable cache" 체크

3. **네트워크 스로틀링** (선택)
   - Network 탭 → "No throttling" 드롭다운 → "Slow 3G" 선택
   - 느린 환경에서의 성능 시뮬레이션

## 2. 페이지 로드 속도 측정

### 2.1 Network 탭 사용

**목표**: 페이지가 완전히 로드되는 데 걸리는 시간 확인

**단계**:

1. DevTools 열기 (F12)
2. **Network** 탭 클릭
3. 페이지 새로고침 (`Ctrl + R` 또는 `F5`)
4. 하단의 통계 확인

**결과 해석**:

```
┌────────────────────────────────────────────────────────────┐
│  53 requests | 2.1 MB transferred | 4.5 MB resources       │
│  Finish: 1.84 s | DOMContentLoaded: 892 ms | Load: 1.23 s  │
└────────────────────────────────────────────────────────────┘
```

| 지표                 | 의미                  | 좋은 값               |
| -------------------- | --------------------- | --------------------- |
| **requests**         | 총 HTTP 요청 수       | 적을수록 좋음 (< 50)  |
| **transferred**      | 실제 전송된 데이터    | 적을수록 좋음 (< 1MB) |
| **DOMContentLoaded** | HTML 파싱 완료 시간   | < 1초                 |
| **Load**             | 모든 리소스 로드 완료 | < 3초                 |
| **Finish**           | 비동기 요청 포함 완료 | < 5초                 |

### 2.2 워터폴 차트 읽기

Network 탭의 각 요청 옆에 있는 막대 그래프:

```
리소스명          │ 워터폴
─────────────────┼──────────────────────────────────
index.html       │ ██                              
app.js           │    ████████                    
style.css        │    ███                          
api/data         │         ██████████████████     ← 오래 걸리는 API!
image.png        │              ████
```

**색상 의미**:

- 🟢 **녹색 (TTFB)**: 서버 응답 대기 시간
- 🔵 **파란색**: 다운로드 시간
- ⚪ **흰색 공백**: 대기 시간 (다른 요청 기다림)

### 2.3 느린 리소스 찾기

1. Network 탭에서 **Time** 컬럼 클릭 (정렬)
2. 가장 오래 걸리는 리소스 확인
3. 해당 리소스 클릭 → **Timing** 탭 확인

## 3. 런타임 성능 분석 (스크롤, 애니메이션)

### 3.1 Performance 탭 사용

**목표**: 페이지 사용 중 버벅거림(프레임 드롭) 감지

**단계**:

1. DevTools 열기 (F12)
2. **Performance** 탭 클릭
3. ⚫ **녹화 버튼** 클릭 (또는 `Ctrl + E`)
4. 페이지에서 **스크롤, 클릭, 드래그** 등 조작
5. ⬛ **정지 버튼** 클릭
6. 결과 분석

### 3.2 결과 화면 이해

```
┌─────────────────────────────────────────────────────────┐
│ ████████░░░░████████████░░░████████████████████████     │ ← 프레임
│ ───────────────────────────────────────────────────     │   (초록=60fps)
│ ┌─────┐ ┌──────────────┐ ┌────────────────┐             │
│ │Task │ │   Task       │ │    Task        │             │ ← JS 실행
│ └─────┘ └──────────────┘ └────────────────┘             │
├─────────────────────────────────────────────────────────┤
│ Summary | Bottom-Up | Call Tree | Event Log             │ ← 분석 탭
└─────────────────────────────────────────────────────────┘
```

### 3.3 주요 지표

| 지표                        | 의미                     | 좋은 값       |
| --------------------------- | ------------------------ | ------------- |
| **FPS** (Frames Per Second) | 초당 프레임 수           | 60 fps (녹색) |
| **Long Task**               | 50ms 이상 걸리는 JS 작업 | 없을수록 좋음 |
| **Total Blocking Time**     | 메인 스레드 차단 시간    | < 200ms       |

### 3.4 문제 찾기

**프레임 드롭 확인**:

- 상단 프레임 바에서 **빨간색 막대** = 프레임 드롭
- 해당 시점 클릭 → 아래에서 원인 확인

**긴 작업 찾기**:

1. Summary 탭에서 "Long Tasks" 확인
2. 빨간 모서리가 있는 작업 = 50ms 초과

### 3.5 실시간 FPS 모니터

1. DevTools → `Ctrl + Shift + P` (명령 팔레트)
2. "Show fps" 입력 → "Show frames per second (FPS) meter" 선택
3. 화면 좌측 상단에 FPS 미터 표시

```
┌────────────┐
│ 60 fps     │
│ ██████████ │
│ GPU: 45%   │
└────────────┘
```

## 4. 메모리 사용량 확인

### 4.1 Memory 탭 사용

**목표**: 메모리 누수 감지, 과도한 메모리 사용 확인

### 4.2 힙 스냅샷 찍기

**단계**:

1. DevTools 열기 (F12)
2. **Memory** 탭 클릭
3. "Heap snapshot" 선택
4. **Take snapshot** 클릭
5. 결과 확인

**결과 해석**:

```
┌─────────────────────────────────────────────────────────┐
│ Snapshot 1                                     25.3 MB  │
│ ├─ (string)                                    8.2 MB   │
│ ├─ (array)                                     5.1 MB   │
│ ├─ (system)                                    4.8 MB   │
│ ├─ Object                                      3.2 MB   │
│ └─ ...                                                  │
└─────────────────────────────────────────────────────────┘
```

### 4.3 메모리 누수 찾기 (비교 방법)

1. 페이지 로드 직후 **Snapshot 1** 찍기
2. 페이지 조작 (데이터 로드, 탭 전환 등)
3. **Snapshot 2** 찍기
4. Snapshot 2 선택 → 드롭다운에서 **"Comparison"** 선택
5. **#Delta** 컬럼 확인 (증가한 객체 수)

**주의할 점**:

- Delta가 계속 증가하면 = 메모리 누수 가능성
- 특히 Detached DOM 요소 확인

### 4.4 실시간 메모리 모니터

1. Performance 탭 → ⚙️ 설정 → "Memory" 체크
2. 녹화 시 메모리 사용량 그래프 표시

또는:

1. DevTools → `Ctrl + Shift + P`
2. "Show performance monitor" 선택
3. 실시간 메모리/CPU 그래프 표시

### 4.5 메모리 사용량 기준

| 앱 규모         | 적정 힙 크기 |
| --------------- | ------------ |
| 단순 페이지     | < 10 MB      |
| 일반 SPA        | 10-50 MB     |
| 복잡한 대시보드 | 50-150 MB    |
| **주의 필요**   | > 200 MB     |

## 5. API 응답 시간 측정

### 5.1 Network 탭에서 API 필터링

**단계**:

1. DevTools 열기 (F12)
2. **Network** 탭 클릭
3. 필터에서 **Fetch/XHR** 클릭 (API만 표시)
4. 페이지 새로고침 또는 조작

### 5.2 API 요청 상세 분석

특정 API 클릭 후:

```
┌─────────────────────────────────────────────────────────┐
│ Headers | Preview | Response | Initiator | Timing       │
└─────────────────────────────────────────────────────────┘
```

**Timing 탭 상세**:

```
Queueing          :   0.5 ms   ← 대기열 대기
Stalled           :   2.1 ms   ← 연결 대기
DNS Lookup        :   0.0 ms   ← DNS 조회
Initial connection:  25.3 ms   ← TCP 연결 (첫 요청만)
SSL               :  15.2 ms   ← HTTPS 핸드셰이크
Request sent      :   0.2 ms   ← 요청 전송
Waiting (TTFB)    : 145.8 ms   ← ⭐ 서버 처리 시간
Content Download  :  12.4 ms   ← 응답 다운로드
─────────────────────────────────
Total             : 201.5 ms
```

### 5.3 주요 API 성능 기준

| 지표                 | 좋음    | 보통       | 나쁨     |
| -------------------- | ------- | ---------- | -------- |
| **TTFB** (서버 응답) | < 100ms | 100-500ms  | > 500ms  |
| **Total Time**       | < 200ms | 200-1000ms | > 1000ms |

### 5.4 SmartFactoryLogger 주요 API

| 엔드포인트              | 용도          | 권장 응답시간 |
| ----------------------- | ------------- | ------------- |
| `/api/data`             | 실시간 데이터 | < 100ms       |
| `/api/config`           | 설정 로드     | < 200ms       |
| `/api/spot/proxy_image` | SPOT 이미지   | < 500ms       |
| `/api/health`           | 건강 체크     | < 50ms        |

### 5.5 느린 API 찾기

1. Network 탭 → **Time** 컬럼 클릭 (정렬)
2. 가장 오래 걸리는 API 확인
3. 반복 호출되는 API 확인 (같은 URL 여러 번)

## 6. 종합 점검: Lighthouse

### 6.1 Lighthouse란?

Google에서 만든 **자동화된 웹사이트 품질 검사 도구**입니다. 성능, 접근성, SEO
등을 종합적으로 점수화합니다.

### 6.2 실행 방법

1. DevTools 열기 (F12)
2. **Lighthouse** 탭 클릭
3. 카테고리 선택:
   - ☑️ Performance (성능)
   - ☑️ Accessibility (접근성)
   - ☐ Best Practices
   - ☐ SEO
4. Device: **Desktop** 선택
5. **Analyze page load** 클릭
6. 약 30초~1분 대기

### 6.3 결과 해석

```
┌─────────────────────────────────────────────────────────┐
│                    Performance: 85                       │
│                    ████████████████░░░░                  │
│                                                          │
│  First Contentful Paint     1.2 s   ██████░░░░           │
│  Largest Contentful Paint   2.1 s   █████████░           │
│  Total Blocking Time        150 ms  ██░░░░░░░░           │
│  Cumulative Layout Shift    0.05    █░░░░░░░░░           │
│  Speed Index                1.8 s   ██████░░░░           │
└─────────────────────────────────────────────────────────┘
```

### 6.4 핵심 지표 (Core Web Vitals)

| 지표                         | 약자 | 좋음    | 개선 필요 | 나쁨    |
| ---------------------------- | ---- | ------- | --------- | ------- |
| **Largest Contentful Paint** | LCP  | < 2.5s  | 2.5-4s    | > 4s    |
| **First Input Delay**        | FID  | < 100ms | 100-300ms | > 300ms |
| **Cumulative Layout Shift**  | CLS  | < 0.1   | 0.1-0.25  | > 0.25  |

### 6.5 점수 기준

| 점수   | 등급    | 의미           |
| ------ | ------- | -------------- |
| 90-100 | 🟢 좋음 | 최적화됨       |
| 50-89  | 🟠 보통 | 개선 여지 있음 |
| 0-49   | 🔴 나쁨 | 즉시 개선 필요 |

### 6.6 개선 제안 활용

Lighthouse는 구체적인 개선 방법도 제시합니다:

```
Opportunities (기회)
──────────────────────────────────────
⚠️ Reduce unused JavaScript          │ 1.2 s  │ 234 KB
   Remove unused code or use code splitting

⚠️ Serve images in next-gen formats  │ 0.8 s  │ 156 KB
   Use WebP instead of PNG/JPG
```

## 7. 실전 체크리스트

### 7.1 빠른 점검 (5분)

- [ ] Network 탭에서 총 로드 시간 확인 (< 3초)
- [ ] 가장 느린 리소스 3개 확인

### 7.2 상세 점검 (30분)

- [ ] 모든 API 응답 시간 확인 (< 500ms)
- [ ] Performance 녹화로 프레임 드롭 확인

- [ ] `/api/data` 폴링 주기 확인 (1초)
- [ ] SPOT 이미지 로드 시간 (< 500ms)
- [ ] 대시보드 위젯 드래그 시 프레임 드롭 없음
- [ ] 설정 창 열기/닫기 시 버벅임 없음

## 8. 자주 묻는 질문

### Q1: 개발 모드 vs 프로덕션 모드?

**항상 프로덕션 빌드로 테스트하세요!**

```powershell
# 프로덕션 빌드
npm run build

# 프로덕션 서버로 실행
npm run preview
# 또는 배포된 EXE 사용
```

개발 모드는 디버깅 코드가 포함되어 훨씬 느립니다.

### Q2: 측정할 때마다 결과가 다른데요?

정상입니다. 더 정확한 결과를 위해:

1. 시크릿 모드 사용
2. 다른 탭/프로그램 닫기
3. 3회 측정 후 평균 사용

### Q3: 점수가 낮으면 어떻게 해야 하나요?

Lighthouse의 **Opportunities** 섹션을 확인하세요. 가장 영향이 큰 항목부터
개선합니다.
