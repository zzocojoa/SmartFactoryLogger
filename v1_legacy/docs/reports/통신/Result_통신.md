# Result_통신

본 문서는 `docs/reports/통신` 폴더 내의 분석 및 보고서를 바탕으로 **수행된
내용**과 **수행되지 않은 내용**을 정리합니다.

## 1. 수행된 내용 (Performed)

### 1.1 통신 데이터 요청 최적화 (Data Diet / Mini-Block Strategy)

- **배경**:
  - 기존 `Merge Mode`가 불필요하게 큰 데이터 블록(약 814 Words)을 요청함.
  - **가설 검증 결과**: 통신 주기를 **0.9초**로 늦춰도 실패가 지속됨을 확인.
    이에 따라 장애 원인이 '통신 속도(Interval)'가 아닌 **'거대 패킷 크기(Giant
    Block)'** 자체에 있음을 규명.
- **수행 내용**:
  - **Mini-Block 전략 적용**: 800워드급 거대 요청을 제거하고, 핵심 데이터만
    핀포인트로 조회하는 5개의 소형 블록(총 55 Words) 요청으로 변경
    (`_read_optimized`).
  - **성과**: 요청 페이로드 크기 **92% 이상 감소**, 장비 부하 최소화 및 0.2초
    주기 안정성 확보.

### 1.2 통신 지연 최소화 (Latency Reduction)

- **TCP_NODELAY 적용**: Extruder와 LS PLC 소켓 연결 시 `Nagle 알고리즘` 비활성화
  옵션 적용. 패킷 버퍼링 없이 즉시 전송하여 실시간성 강화.

### 1.3 장애 대응 및 회복 로직 (Skip Logic)

- **Error Cooldown 도입**: 통신 에러 발생 시 즉시 재시도하거나 주기를 늘리는
  대신, **지정된 횟수(5회, 약 1초)만큼 요청을 생략(Pass)**하는 로직 적용.
  - 0.2초 고정 주기의 시계열 박자(Rhythm)를 깨지 않으면서 장비가 회복할 시간을
    부여.

### 1.4 안정성 강화

- **루프 안전 처리**: PLC 응답 패킷 파싱 시 빈 값이나 예외 상황에 대한 방어 코드
  추가 (프로그램 셧다운 방지).
- **타임아웃 현실화**: 0.15초 등 비현실적인 타임아웃 대신 `0.5s` 이상의 안정적
  타임아웃 유지.
- **로그 및 파일 I/O 안정화**:
  - 로그 경로 유효성 검증 및 로컬 경로(`APP_DATA_DIR/logs`) Fallback 구현.
  - 파일 쓰기 스레드 예외 격리 및 재시도 로직으로 데이터 수집 중단 방지.

## 2. 수행되지 않은 내용 / 보류 사항 (Not Performed)

### 2.1 불안정하거나 제약 사항 위배로 폐기된 기술

- **Circuit Breaker (가변 주기)**: 에러 시 수집 주기를 0.2초 → 0.5초로 늘리는
  안은 **"0.2초 주기 고정"** 핵심 요구사항 위배로 폐기.
- **Short Socket Timeout (0.15s)**: 너무 짧은 타임아웃은 구형 장비의 정상
  응답조차 에러로 처리할 위험이 있어 폐기.
- **TCP Keep-Alive**: 0.2초마다 데이터를 주고받는 상황에서 1초 단위 Keep-Alive는
  트래픽 낭비로 판단하여 미적용.
- **Smart Split (Mini-Batch)**: 실패 후 중간 크기로 재시도하는 복잡한 로직 대신,
  아예 처음부터 최적화된 크기로 요청하는 **Mini-Block 전략(1.1)**이 채택되어
  폐기.

### 2.2 향후 과제 / 미진행 사항

- **네트워크 품질 정량적 데이터 확보**: 장기간의 RTT 측정, 패킷 손실률 등
  네트워크 품질에 대한 정밀 진단 리포트는 작성되지 않음 (현상 해결 우선).
- **성능 지표 대시보드**: 통신 성공률, 평균 응답 속도 등을 실시간 시각화하는
  기능은 코드에 반영되지 않음.
- **시작 버튼 제어**: 통신 스레드 중복 실행 방지를 위한 UI 레벨(버튼 비활성화)
  처리는 미구현 (백엔드 락으로 방어 중).
