# 사용자 설정 관리 방식 비교 리포트

## 1. 비교 대상
- `docs/reports/운영/상용_배포_사용자_설정_관리_리포트.md` (INI 기반, 현장 이식 PC 중심 운영)
- `docs/reports/운영/json_사용자_설정_관리.md` (JSON 기반 일반 가이드 및 예제)

## 2. 핵심 결론
- 본 프로젝트(공장 현장 PC 이식, `config.ini` 기준)에는 INI 방식이 운영 적합성이 더 높다.
- JSON 방식은 구조화/검증/버전 관리에 강점이 있어 고급·확장 설정에 보조로 결합하는 것이 최적이다.

## 3. 방식별 비교
- 형식/구조: INI는 단순한 섹션/키 구조로 빠른 편집이 가능하고, JSON은 계층 구조로 복잡한 설정에 유리하다.
- 운영자 접근성: INI가 현장 작업자에게 직관적이며, JSON은 문법 오류에 취약해 교육/도구가 필요하다.
- 저장 위치/이식성: INI는 이식 PC의 `config.ini`를 단일 기준으로 삼기 쉽고, JSON은 OS 권장 경로(APPDATA 등) 활용에 적합하다.
- 안정성/검증: JSON 문서가 제시한 원자적 저장, 스키마 검증, 마이그레이션 로직은 INI에도 동일하게 적용되어야 한다.
- 주석/설명: JSON은 주석 불가로 `_comment` 같은 우회가 필요하고, INI는 주석이 쉬워 현장 가이드에 유리하다.

## 4. 결합 권장안
- 기본 운영값(접속 IP/포트, 수집 주기 등)은 `config.ini`에 유지하고 이식 PC 기준을 고정한다.
- 고급/확장 설정(UI, 장비별 상세, 화면 구성 등)은 JSON으로 분리해 구조화한다.
- 우선순위 규칙: 이식 PC `config.ini` > JSON > 코드 기본값.
- 공통 운영 원칙: 스키마 검증, 원자적 저장, 백업/복구 정책, 버전 필드와 마이그레이션 로직을 통일한다.

## 5. 추가 보완 제안
- 인코딩: `config.ini`는 UTF-8 with BOM 생성, 로딩 시 `utf-8-sig` 우선 + `cp949` fallback으로 현장 메모장 호환성 확보.
- 환경 변수 우선: 자동 배포/컨테이너 대비로 `ENV > config.ini > 기본값` 규칙을 명문화하고 덮어쓴 값은 로그로 추적.
- 읽기 전용 예외: `PermissionError` 발생 시 명확한 UI 안내와 임시 저장/재시도 흐름을 제공.

## 6. 프로젝트 적용 판단
- 현 단계에서는 INI 단독 방식으로도 충분하며 운영 리스크가 낮다.
- 설정 항목이 확대되면 JSON 병행 도입이 장기 유지보수에 유리하다.
