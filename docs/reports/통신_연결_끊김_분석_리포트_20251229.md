# 압출기/PLC 연결 끊김 통합 분석 리포트

## 1. 요약
약 2주간의 시범 운영 중 압출기(Extruder)와 PLC 통신이 간헐적으로 끊기며 데이터 수집이 중단되는 현상이 관찰되었습니다. 제공된 로그는 원격 호스트가 연결을 강제로 종료한 상황(WinError 10054)을 보여주며, 현장 네트워크 품질, 장비 측 세션 정책, 그리고 클라이언트 수신 처리 방식이 복합적으로 영향을 미쳤을 가능성이 높습니다. 또한 PLC 오류 직후 `KeyError: 'Mold1'`로 데이터 수집 스레드가 중단된 사례가 확인되었습니다.

## 2. 사건(Incident) 개요
- 기간: 시범 운영 약 2주
- 증상: 간헐적으로 압출기/PLC 연결이 끊기고 데이터 수집이 끊김
- 로그(제공):
  - `[2025-12-29 23:04:41] [ERROR] [Extru] IO Error on D0020: [WinError 10054] 현재 연결은 원격 호스트에 의해 강제로 끊겼습니다`
  - `[2025-12-29 23:04:41] [ERROR] [PLC] Data/Validation Error: [WinError 10054] 현재 연결은 원격 호스트에 의해 강제로 끊겼습니다`
  - `[2025-12-29 23:04:41] [CRITICAL] Data Loop Error: 'Mold1'` (KeyError로 수집 루프 중단)

### 2.1 타임라인 및 인과관계
- 23:04:41: 압출기/PLC 연결이 동시에 강제 종료(WinError 10054)로 로그 생성.
- 동일 시각: PLC 오류 후 응답 데이터가 `{}`로 반환되며 `KeyError: 'Mold1'` 발생.
- 결과: `data_collection_loop`에서 예외 발생 후 스레드 종료.

### 2.2 원인 분류(1차/2차)
- 1차 원인: 원격 호스트가 연결을 강제 종료(WinError 10054).
- 2차 원인: PLC 오류 경로에서 `{}` 반환 후 키 직접 접근으로 `KeyError: 'Mold1'` 유발.
- 최종 결과: 데이터 수집 스레드 중단.

### 2.3 현장 system.log 분석 요약
- WinError 10054: Extru 2건, PLC 2건 (2025-12-10 12:34:50, 2025-12-29 23:04:41).
- Queue full 경고 30,360건: 2025-12-10 18:51:46 ~ 2025-12-22 10:23:53.
- file_writer_thread Invalid argument 9건: 2025-12-10 12:34:51 ~ 2025-12-15 14:45:45.
- CSV 로그 파일 오픈 실패 6건: 2025-12-10 12:34:51 ~ 12:34:57 (Z: 경로 문자열 깨짐).
- Data collection thread started 중복 1회: 2025-12-23 09:10:28 (동일 시각 2회).

## 3. 기술적 관찰
### 3.1 공통 운영 조건
- 수집 주기: 이식된 컴퓨터의 `config.ini` 기준 `intervalsec = 0.2` (0.2초/5Hz).
- 장비 3종(Extruder, PLC, SPOT)을 병렬로 폴링.

### 3.2 압출기 통신 특성
- 한 주기에서 압출기 레지스터를 5회 읽음(D0020, B1502, D0420, D1500, D1900).
- 결과적으로 초당 약 25회 TCP 요청/응답 발생.
- 소켓 타임아웃 0.15초로 매우 짧음.
- `recv(4096)` 단발 호출로 수신 완전성 보장 로직 부재.

### 3.3 PLC 통신 특성
- 매 주기 멀티블록 요청 1회 수행.
- 소켓 타임아웃 0.5초 설정.
- `recv(4096)` 단발 호출로 패킷 길이(body_len)만큼 수신 보장하지 않음.
- 예외 발생 시 `{}` 반환 후 소켓 종료.

## 4. 추정 원인(가설)
H1) **장비 측 세션 정책/부하로 인한 연결 종료** (신뢰도: 중~상)
- 고빈도 요청(0.2초 주기)과 반복 폴링이 장비 측 제한을 초과했을 가능성.

H2) **네트워크 품질 저하 또는 장비 재시작** (신뢰도: 중)
- 현장 네트워크 지터, 스위치/케이블 이슈, 장비 리셋이 10054 오류를 유발할 수 있음.

H3) **부분 수신으로 인한 프로토콜 비동기** (신뢰도: 중)
- 단발 `recv()`로 패킷이 완전히 수신되지 않으면 파싱 실패가 발생할 수 있음.
- 반복 실패 시 장비가 연결을 강제로 종료할 가능성.

## 5. 영향 분석
- 압출기/PLC 데이터가 누락되어 대시보드 및 로그 품질 저하.
- PLC 오류 발생 시 `{}` 반환이 상위 루프의 KeyError로 이어져 수집 스레드가 중단됨.
- Queue full 경고가 장기간 반복되어 Ring Buffer가 오래된 데이터를 지속적으로 드롭함.
- 파일 기록 오류(Invalid argument, CSV 오픈 실패)로 로그 저장이 누락/지연되고 큐 적체가 심화됨.
- 수집 스레드 시작 로그가 빈번하며 동일 시각 중복 시작 1회가 확인됨(중복 실행/재시작 가능성).

### 5.1 KeyError 'Mold1' 발생 경로
- `src/modules/ls_plc.py`에서 예외 시 `{}`를 반환.
- `src/main.py`가 `ls['Mold1']` 등 키를 직접 접근.
- PLC 통신 오류와 동일 타임스탬프에 KeyError가 발생하며 수집 루프가 종료됨.

## 6. 개선 제안
### 6.1 즉시 적용 권장
1) 수집 주기 0.2초 유지 전제 확정
- 주기 변경 없이 요청 최적화, 수신 안정화, 큐 처리 개선으로 부하를 낮춤.

2) 상위 루프 안전 처리
- PLC 응답이 빈 딕셔너리여도 루프가 지속되도록 안전 접근 로직 추가.
  - 예: `ls.get('Mold1')` 방식 또는 실패 시 기본 키를 포함한 딕셔너리 반환.

3) 타임아웃 상향
- 압출기 타임아웃 0.15초 → 0.5~1.0초로 상향.

4) 로그 경로 유효성 검증 및 로컬 fallback
- 이식 PC에서 `logpath` 유효성 확인 후 실패 시 `./logs`로 자동 전환.

5) 파일 기록 오류 격리/재시도
- file_writer_thread 예외 발생 시 백오프/재시도와 오류 카운트로 지속 수집 보장.

### 6.2 단기 개선
1) 수신 완전성 보장
- 응답 길이(body_len) 또는 종결 문자열 기준으로 `recv()` 루프 구현.

2) 요청 최적화
- 압출기 레지스터 요청을 가능한 범위에서 블록 읽기로 통합.

3) 재연결/백오프 정책
- 연속 실패 시 지수형 backoff로 장비 부하 감소.

4) 큐/버퍼 처리량 개선
- Ring Buffer 용량 상향, 배치 저장, 큐 길이 모니터링으로 데이터 드롭 최소화.

5) 중복 실행 방지
- 단일 실행 락 및 시작 버튼 비활성화로 중복 수집 스레드 방지.

### 6.3 중기 개선
1) 네트워크 품질 관측
- 스위치 포트 에러율, 링크 리셋 횟수, 케이블 상태 점검.

2) 장비 통신 스펙 확인
- 제조사 권장 요청 빈도, 동시 세션 수 제한 확인 및 정책 반영.

## 7. 검증 방법
- 0.2초 주기 유지 상태에서 요청 최적화/수신 루프 적용 전후 10054 발생 빈도 비교.
- 수신 루프 적용 후 파싱 오류/연결 종료 감소 여부 측정.
- 현장 스위치/케이블 점검 전/후 재발률 비교.
- logpath 유효성/로컬 fallback 적용 후 CSV 오픈 실패 및 Invalid argument 0건 확인.
- Queue full 경고 건수(30,360 기준)가 유의미하게 감소했는지 비교.
- 동일 시각 중복 시작 로그가 재발하지 않는지 확인.

### 7.1 성능 지표(0.2초 고정 전제)
- 통신 실패율: 장비별 `WinError 10054`/총 요청 수.
- 데이터 유실: Queue full 발생 건수 및 CSV 누락율(기대 행 수 대비).
- 파일 기록 품질: `Invalid argument`/CSV 오픈 실패 건수.
- 처리 지연: 실제 루프 주기 편차 및 지연 누적 시간.

### 7.2 실측 계획
1) 변경 전 기준선 24~48시간 수집(system.log/CSV/OS 성능).
2) 변경 후 동일 조건 재측정(0.2초, 동일 라인/네트워크).
3) 전후 비교표로 개선 여부 판단(10054 0~1회/주, Queue full 감소, 기록 오류 0건).

## 9. 결론
WinError 10054는 장비 측 세션 정책/부하 또는 네트워크 불안정, 그리고 부분 수신 처리 문제의 결합으로 발생했을 가능성이 높습니다. system.log 분석 결과 10054 자체는 2회로 드물었으나, Queue full 경고(30,360건)와 파일 기록 오류가 장기간 발생해 데이터 손실이 누적되었습니다. 운영 안정성 확보를 위해 수집 주기 0.2초 고정 조건에서 요청 최적화와 수신 완전성 보장이 우선이며, PLC 오류 시 상위 루프가 중단되지 않도록 안전 접근을 추가하는 것이 필수입니다. 또한 logpath 검증/로컬 전환과 file_writer_thread 처리량 개선을 병행해야 하며, 이후 장비 제조사 스펙에 맞춘 통신 정책 정립과 네트워크 품질 점검을 이어가야 합니다.
