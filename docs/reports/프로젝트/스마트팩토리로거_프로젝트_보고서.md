# 스마트팩토리로거 프로젝트 보고서

## 1. 요약
본 프로젝트는 알루미늄 압출 라인의 현장 데이터를 수집하고 운영자가 즉시 이해할 수 있도록 시각화하는 산업용 데이터 로거 및 HMI입니다. 압출기 컨트롤러, LS XGT PLC, SPOT 적외선 온도계/카메라를 통합하여 실시간 모니터링과 CSV 로깅을 수행하며, 현장 운영을 고려한 안정성(상태 복구, 크래시 로그)과 배포 편의성(포터블 실행 파일)을 갖춘 실무형 코드베이스입니다.

## 2. 목적 및 범위
- 목적: 공정 데이터를 지속적으로 수집하여 운영/분석에 바로 활용 가능하게 만드는 것.
- 범위: 장비 통신, 실시간 GUI, 로깅, 설정 관리.
- 운영 환경: 공장 현장의 간헐적 통신 장애와 간단한 배포/복구 요구를 전제로 설계.

## 3. 아키텍처 개요
### 3.1 런타임 오케스트레이션
- 진입점 `src/main.py`가 크래시 로깅, 시그널 처리, 작업 스레드, UI를 초기화.
- `APP_DATA_DIR/app.lock`로 단일 실행 보장.
- `ThreadPoolExecutor`로 압출기/PLC/pyrometer를 병렬 폴링.
- 로그/UI는 `collections.deque(maxlen=5000)` 링버퍼로 분리하여 생산자-소비자 병목을 완화.
- 큐 포화 경고는 200회마다 1회로 제한.
- UI가 지연되어도 데이터 수집 스레드는 독립적으로 동작하며, 버퍼 포화 시 오래된 데이터가 순차적으로 삭제되는 구조.
- 수집 주기(0.2초 고정)는 목표 주기이며, 장비 응답 지연 시 실제 주기는 늘어날 수 있음.
- `src/modules/logger.py`의 전용 스레드가 CSV 저장 및 파일 회전을 담당.

### 3.2 장비 인터페이스
- `src/modules/extruder.py`: TCP 기반 레지스터 읽기 프로토콜 사용. (recv loop, 블록 통합/분할 재시도, backoff).
- `src/modules/ls_plc.py`: LS XGT 멀티블록 읽기 패킷 구성 및 응답 파싱(통신 부하 저감). (길이 기반 recv, 오류 시 기본 dict, backoff).
- `src/modules/spot.py`: HTTP GET으로 SPOT 온도 값을 조회.

### 3.3 비즈니스 로직 및 상태
- `src/modules/logic_processor.py`: 센서 신호로부터 die/billet 사이클 ID를 산출.
- 상태는 `state.json`으로 영속화하여 재시작 시 연속성을 확보.

### 3.4 GUI 및 시각화
- `src/gui.py`: KPI 카드/게이지/알림 기반의 운영자 대시보드.
- `src/modules/graph_view.py`: 시계열 그래프, 임계치 라인, 스냅샷 저장.
- `src/modules/threshold_gui.py`: 임계치 설정과 `config.ini` 반영.
- `src/settings_gui.py`: 장비 엔드포인트와 로깅 경로를 현장에서 변경 가능.

### 3.5 설정 관리
- `src/config.py`: 포터블/표준 설치 경로 탐지 및 기본값 병합.
- `src/config_schema.py`: Pydantic 기반 설정 검증/정규화.
- `config/config.ini`: 소스 모드에서 현장 설정의 단일 진입점.
- LOG_PATH 유효성 실패 시 `APP_DATA_DIR/logs`로 자동 전환.

## 4. 주요 기능
- 실시간 대시보드(게이지, 카드, 상태 경고) 제공.
- 이종 장비 병렬 폴링으로 서브초 주기 수집.
- CSV 로깅 및 일자/빌렛 기반 회전 로직.
- 크래시 로그 및 상태 복원으로 가동 연속성 확보.
- 임계치 시각화 및 스냅샷 저장 기능.
- 통신 안정화: 수신 루프, 블록 통합/분할 재시도, 백오프.
- LOG_PATH 자동 전환 + 단일 실행 잠금(app.lock)으로 현장 안정성 강화.
- PyInstaller 기반 포터블 배포(`SmartFactoryLogger.spec`).

## 5. 배포 및 운영
- `start.bat`, `start.sh`: 가상환경 생성, 의존성 설치, 실행 자동화.
- `SmartFactoryLogger.spec`: 리소스/설정/스플래시 포함 빌드.
- PyInstaller 기반 포터블 `.exe`는 현장 PC에서 별도 Python 설치 없이 실행 가능하도록 설계.
- `dist/`, `build/`: 빌드 아티팩트로 운영 코드와 분리 필요.
본 프로그램은 개발 환경에서 빌드 후 실제 공장 현장 PC로 이식하여 사용되는 전제를 가집니다. 따라서 `config.ini`와 `src/config.py`의 경로/모드(Portable vs Standard) 설계 및 변경은 현장 이식/운영 시나리오를 기준으로 검토하는 것이 중요합니다. 현장 PC에는 `config.ini`를 동봉 배포하여 IP/포트 변경을 운영자가 즉시 반영할 수 있도록 하는 것이 적합합니다. 이식된 PC의 `config.ini` 기준으로 실행하며, 로그 경로가 유효하지 않으면 `APP_DATA_DIR/logs`로 자동 전환합니다.

## 6. 데이터 및 산출물
- `logs/`, `dist/logs/`: 실제 운영 CSV 로그 포맷 확인 가능.
- `snapshots/`: GUI 그래프 이미지 저장 결과.
- `docs/ametek_land_spot_plus_rest_api_user_guide_issue2_en.pdf`, `logs/SPOT API.txt`: SPOT 장비 프로토콜 문서.
- `tests/`: 로그 분석 및 온도 보정 ML 스크립트 모음.

## 7. 강점
- 수집/로깅/GUI 역할 분리가 명확하여 안정적.
- 동시성 구조로 UI 정지 없이 고주기 수집 유지.
- 크래시 로그/상태 복구/로그 회전 등 실무 안정성 고려.
- 운영자 중심의 경고, 진단, 빠른 설정 접근 제공.
- 통신 안정화 개선(수신 루프, 백오프, 블록 통합/분할 재시도).
- 단일 실행 잠금과 LOG_PATH 자동 전환으로 현장 운영 안정성 강화.
## 8. 리스크 및 이슈(실무 관점)
R1) 런타임 의존성 누락 (우선순위: P0)
   - `pydantic`이 `requirements.txt`에 없음.
   - 결과: 신규 설치 시 기동 실패 가능.

R2) GUI 경고 타이머 오류 (우선순위: P1)
   - `src/gui.py`에서 `datetime.timedelta` 사용 시 `timedelta` 미임포트.
   - 결과: 경고 표시 시 런타임 오류 가능.

R3) 설정 UI 로그 뷰어 오류 (우선순위: P2)
   - `src/settings_gui.py`에서 `BASE_DIR` 미정의.
   - 결과: 상대 경로 로그 설정 시 NameError.

R4) PLC 예외 경로에서 데이터 루프 중단 가능 (우선순위: P0) (해결)
   - `src/modules/ls_plc.py`가 예외 시 `{}` 반환, `src/main.py`는 키 직접 접근.
   - 결과: KeyError로 데이터 루프 중단 가능. (해결)
   - 조치: 오류 시 기본 dict 반환 + main 안전 접근.

R5) 소켓 수신 단발 처리 (우선순위: P1) (해결)
   - `src/modules/ls_plc.py`, `src/modules/extruder.py`가 `recv()` 단발 호출. (해결)
   - 조치: 수신 루프 + 길이 기반 수신으로 보완.
   - 결과: 네트워크 지터에서 패킷 누락/파싱 오류 가능.

R6) GUI 하드코딩 URL (우선순위: P2)
   - `src/gui.py`에서 카메라/액추에이터 링크 IP가 고정.
   - 결과: 현장 설정 변경 시 UI와 실제 설정 불일치.

R7) 유지보수성 부채 (우선순위: P2)
   - `src/config.py`에 중복 기본값(`RefreshInterval`).
   - `src/settings_gui.py`에 `on_close` 중복 정의.
   - `src/gui.py`에 중복/불필요 임포트 존재.

## 9. 개선 제안
### 9.1 즉시 수정
- `requirements.txt`에 `pydantic` 추가.
- `src/gui.py`에 `timedelta` 임포트 또는 `datetime` 모듈 전체 임포트.
- `src/settings_gui.py`에서 `BASE_DIR` 참조 경로 정리.
- `src/main.py`의 PLC 데이터 접근에 안전 처리 추가. (완료)

### 9.2 단기 개선
- 압출기 레지스터 매핑을 외부 설정으로 분리.
- GUI 하드코딩 URL을 설정 기반으로 전환.
- 소켓 수신을 길이 기반으로 재구성하여 부분 수신 문제 최소화. (완료)
- PLC/Extruder 프로토콜 파싱에 대한 Mock 서버 기반 테스트 도입.
- 큐 드롭/포화 지표와 알림 추가.
- 재시도/블록 통합 정책 임계값을 설정으로 외부화.
### 9.3 중기 개선
- 시계열 DB(SQLite/InfluxDB) 도입으로 조회/분석 효율 개선.
- 장비 장애 알림을 별도 채널로 노출해 운영 중단 최소화.
- 설정 마이그레이션/검증 로직의 통합 모듈화.

## 10. 테스트 메모
- `tests/` 내 ML/분석 스크립트는 런타임과 별도의 의존성(pandas, sklearn, joblib)이 필요하므로, 운영 환경과 분리된 실행 지침이 필요합니다.

## 11. 결론
SmartFactoryLogger는 현장 운영을 고려한 실무형 로거로서 기본 안정성과 UI 완성도가 높습니다. 다만, 의존성 누락과 예외 경로 안정성 같은 즉시 수정 항목을 빠르게 보완하면 운영 안정성이 크게 개선됩니다. 이후 설정 외부화와 테스트 체계 보강을 진행하면 장기 유지보수성도 충분히 확보할 수 있습니다.
