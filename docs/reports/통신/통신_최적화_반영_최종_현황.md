# 통신 성능 최적화 반영 최종 현황

`수정 일자: 2025-12-31`

본 문서는 `통신_성능_최적화_구현_계획.md`을 기반으로 실제 코드에 **반영된
항목**과 검토 과정에서 **제외된 항목**을 정리합니다.

## 1. 반영된 핵심 기술 (Implemented)

### 1.1 TCP_NODELAY (Nagle Logic Off)

- **적용 위치**: `extruder.py`, `ls_plc.py`
- **내용**: `sock.setsockopt(socket.IPPROTO_TCP, socket.TCP_NODELAY, 1)` 적용.
- **반영 이유**: 0.2초라는 짧은 주기 내에서 작은 데이터 패킷이 버퍼링 되지 않고
  **즉시 전송**되도록 하여 불필요한 레이턴시를 제거함. 부작용이 없는 가장 안전한
  최적화.

### 1.2 Skip Logic (Error Cooldown)

- **적용 위치**: `extruder.py`
- **내용**: 통신 에러 발생 시 `skip_counter = 5` (약 1초) 설정.
- **반영 이유**:
  - **0.2초 고정 주기 준수**: 주기를 가변적으로 늘리지 않고, 메트로놈
    박자(0.2s)는 유지하되 "이번 박자는 쉰다"는 개념 적용.
  - **과부하 방지**: 에러 후 무리한 즉시 재접속 시도 대신 장비 소켓이 자연
    회복될 시간을 보장.

## 2. 반영되지 않은 항목 (Discarded) & 사유

### 2.1 Circuit Breaker (가변 주기)

- **내용**: 에러 시 수집 주기를 0.2초 → 0.5초로 일시 변경.
- **미반영 사유**: **"0.2초 주기 고정"**이라는 사용자의 핵심 요구사항과 충돌.
  데이터 시계열 정합성을 위해 가변 주기 대신 `Skip Logic`(정해진 횟수만큼
  건너뛰기)으로 대체함.

### 2.2 Short Socket Timeout (0.15s)

- **내용**: 타임아웃을 수집 주기보다 짧게 설정.
- **미반영 사유**: 구형 PLC/압출기의 응답 속도를 고려할 때 너무 공격적인 설정임.
  정상적인 응답조차 타임아웃으로 처리하여 재연결 폭주를 유발할 위험이 있어
  **기존 보수적 설정 유지**.

### 2.3 TCP Keep-Alive

- **내용**: 1초 단위의 Keep-Alive 패킷 전송.
- **미반영 사유**: 0.2초마다 데이터를 주고받는 활성(Active) 상태에서는
  무의미하며, 오히려 불필요한 트래픽 부하만 가중시킴.

### 2.4 Smart Split (Mini-Batch)

- **내용**: Read Block 실패 시 중간 크기(Mini-Batch)로 재시도.
- **미반영 사유**: 장비별 호환성 검증이 안 된 상태에서 로직 복잡도만 높임.
  단순하고 확실한 **Merge(전체) ↔ Split(개별) 이원화 유지**가 더 안정적이라
  판단.

## 3. 결론

이론적인 최적화 기법들 중 **"0.2초 주기를 엄수"**하면서도 **"실무 환경에서 가장
안전하고 효과적인"** 두 가지(NODELAY, Skip Logic)만 선별하여 적용했습니다. 이를
통해 **반응 속도는 높이고 시스템 부하는 낮추는** 균형 잡힌 튜닝이
완료되었습니다.
