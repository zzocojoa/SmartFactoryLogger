# 통신 성능 최적화 및 안정화 구현 계획 (Final)

## 1. 개요

본 문서는 수집 주기 **0.2초(5Hz)**를 엄격히 준수하면서, `WinError 10054` 및
과부하 문제를 해결하기 위한 **현실적이고 안전한** 코드 수정 계획입니다. 이전
검토에서 제기된 "오버 엔지니어링(Keep-Alive, 복잡한 TimeOut 등)" 요소를 전면
배제하고, **검증된 핵심 기술**만 적용합니다.

## 2. 목표과 원칙

1. **원칙 1**: 0.2초 수집 주기는 절대 변경하지 않는다. (가변 주기 금지)
2. **원칙 2**: 장비(PLC) 특성을 고려하여 불필요한 트래픽(Keep-Alive 등)을
   유발하지 않는다.
3. **목표**: `TCP_NODELAY`로 반응 속도를 높이고, 에러 발생 시 부하를 주는 재시도
   대신 **쿨다운(Skip)** 전략을 사용한다.

## 3. 상세 구현 내용

### 3.1 [Core] TCP_NODELAY 적용 (확정)

- **설명**: Nagle 알고리즘(패킷 모아 보내기)을 끄고 즉시 전송.
- **효과**: 0.2초라는 짧은 주기 내에서 패킷 지연을 줄여 즉각 반응을 돕는다.
  과장 없이 현실적인 개선책으로 적용.
- **적용**: `src/modules/extruder.py`, `src/modules/ls_plc.py`

### 3.2 [Logic] Error Cooldown (Skip Logic) 도입

- **변경 전**: 에러 발생 시 즉시 재시도하거나, 주기에 맞추려 노력함 → 장비에
  연속 타격.
- **변경 후**: "이번 0.2초 사이클이 실패하면, 다음 N번의 사이클은 **요청 자체를
  생략(Skip)**한다."
- **동작 방식**:
  1. 통신 에러 발생 (`WinError` 등).
  2. `skip_counter = 5` (약 1초간 휴식) 설정.
  3. 이후 루프에서는 실제 `send()`를 하지 않고 `return None` 처리하며 카운터
     감소.
  4. 카운터 0이 되면 다시 정상 요청.
  5. 현장 복구 시간에 맞춰 `skip_counter`는 3~10 범위에서 조정 가능.
- **장점**:
  - **0.2초 메트로놈(Grid)**은 유지되므로 데이터 시계열이 깨지지 않음. (단지
    값이 Null인 행이 생길 뿐) 공백 행은 의도된 휴식 신호로 관리하며, system_2.log/CSV에서 확인된 31~32행 공백 패턴과 동일하게 모니터링한다.
  - 장비 입장에서는 1초간 완벽한 휴식을 취하게 되어 TCP 소켓이 자연스럽게
    회복됨.

### 3.3 [Clean-up] 불필요한 기능 제거 (다이어트)

- **삭제**: `SO_KEEPALIVE` (불필요한 트래픽 유발)
- **삭제**: `settimeout(0.15)` (너무 짧음, 오동작 위험) → 기존 기본값 또는 0.5초
  수준 유지.
- **삭제**: `Mini-Batch` (장비 호환성 불확실) → 기존 `Merge` or `Split` 단순
  이원화 유지.

## 4. 작업 순서

1. **`src/modules/extruder.py`**:
   - `TCP_NODELAY` 옵션 추가.
   - `get_data()` 내부에 에러 발생 시 `skip_count` 로직 적용.
2. **`src/modules/ls_plc.py`**:
   - `TCP_NODELAY` 옵션 추가.
3. **검증**:
   - 로그에서 `WinError 10054` 빈도 감소 확인.
   - 장애 시 재연결 폭주 대신 "차분한 1초 공백" 후 복구되는지 확인.

## 5. 결론

가장 **심플(Simple)**한 것이 가장 강력합니다. 복잡한 기교를 부리기보다
기본기(`NODELAY`)와 여유(`Skip`)를 주는 방식으로 안정화를 달성하겠습니다.
