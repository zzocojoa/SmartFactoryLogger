# 통신 장애 원인 재분석 리포트

## 1. 개요

본 문서는 `docs/reports/통신` 디렉토리 내의 기존 분석 문서들과 `system.log`,
소스 코드(`extruder.py`, `ls_plc.py`)를 종합하여 통신 장애(WinError 10054,
데이터 공백)의 원인을 재분석한 결과입니다.

## 2. 현재 상황 요약

- **주요 증상**:
  1. **WinError 10054**: 압출기 및 PLC 통신에서 "원격 호스트에 의해 연결이
     강제로 끊김" 오류 발생.
  2. **데이터 공백**: 오류 발생 시 약 6초(`31~32행` 누락)의 데이터 수집
     공백(Null) 구간 발생.
  3. **블록 병합 실패 가능성**: 로그상 `IO Error on D0020`이 확인되는데, 이는
     `extruder.py`의 **Split Mode(분리 요청 모드)**에서 실행되는 코드입니다.

## 3. 심층 원인 분석

### 3.1 압출기: Split Mode 진입으로 인한 부하 가중

소스 코드(`extruder.py`) 분석 결과, 기본적으로 통신 횟수를 줄이기 위해 대용량
블록 읽기(Merge Mode)를 시도하지만, 실패 시 **Split Mode**로 전환되도록 설계되어
있습니다.

- **Merge Mode (이상적)**: 2회 요청 (Main Block 400워드 + Aux Block 411워드)
- **Split Mode (현재 추정)**: 5회 요청 (D0020, B1502, D0420, D1500, D1900 개별)

**분석 결과**: 로그에 찍힌 `IO Error on D0020` 메시지는 **Split Mode가 동작
중임**을 시사합니다. (Merge Mode에서는 `D0020` 단독 요청을 하지 않음). 즉, 초기
Merge 요청이 실패하여 Split Mode로 진입했고, 이로 인해 주기당 요청 횟수가
**2.5배 증가(2회 → 5회)**했습니다. 0.2초(5Hz) 주기에서 초당 25회의 요청이
발생하며, 이 빈도를 견디지 못한 장비(PLC/압출기 컨트롤러)가 세션을 강제로
종료(10054)하는 악순환이 반복되고 있습니다.

### 3.2 Recovery Time으로 인한 데이터 공백 (6초)

오류 발생 후 약 6초간 데이터가 비는 현상은 소스 코드에 구현된 **Backoff(재시도
지연) 정책**의 결과입니다.

- 연결 끊김(10054) 발생 → `sys_logger.error` 기록
- `_increase_backoff()` 실행: 재연결 대기 시간이 1초 → 2초 → 4초로
  기하급수적으로 증가.
- 이 대기 시간 동안 수집 루프는 돌지만 데이터는 `Null`로 채워지므로, CSV에는
  타임스탬프만 찍히고 값은 없는 행이 기록됩니다.
- **결론**: 6초 공백은 시스템 오류가 아니라 **안전 장치(Backoff)**가 정상 작동한
  결과이나, 근본 원인(연결 끊김)이 해결되지 않아 빈번하게 발생하고 있습니다.

### 3.3 PLC: 0.2초 주기의 한계

PLC 또한 `WinError 10054`와 `Incomplete header response`가 발생합니다. XGT
프로토콜 특성상 요청-응답이 동기식으로 이루어져야 하는데, 0.2초 주기는 네트워크
지터나 PLC 처리 지연 발생 시 **이전 요청 처리가 끝나기 전에 다음 요청이
시도**되거나, 소켓 버퍼 꼬임이 발생하기 쉬운 환경입니다.

## 4. 종합 결론

현재 발생하는 통신 장애의 근본 원인은 **"최적화 모드(Merge) 실패로 인한 고빈도
요청(Split)으로의 전환"**과 **"0.2초라는 극한의 수집 주기"**가 결합된 것입니다.

1. **Merge Mode 실패**: 장비가 한 번에 400워드 이상의 대량 데이터를 보낼 때
   패킷이 잘리거나 타임아웃이 발생하여 Split Mode로 강등됨.
2. **트래픽 폭증**: Split Mode에서 초당 25회 요청 발생.
3. **연결 강제 종료**: 장비가 트래픽 부하를 견디지 못하고 RST(Reset) 패킷
   전송(10054).
4. **데이터 공백**: 소프트웨어는 이를 감지하고 6초간 Backoff 수행.

## 5. 제언 (코드 수정 없음)

현재 상태에서 코드 수정 없이 상황을 완화하려면, 운영 환경에서 **수집
주기(IntervalSec)**를 **0.5초 이상**으로 완화하는 것이 가장 현실적이고 즉각적인
해결책입니다. 0.2초를 고수해야 한다면, Split Mode가 아닌 Merge Mode가 안정적으로
동작하도록 장비의 Max Word 설정을 튜닝하거나 네트워크 환경을 개선해야 합니다.
