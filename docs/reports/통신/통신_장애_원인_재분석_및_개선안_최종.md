# 통신 장애 원인 재분석 및 개선안 (최종)

## 1. 분석 개요

본 리포트는 `system.log` (개선 전), `system_2.log` (개선 후), 그리고
`Factory_Integrated_Log_20251231_000000.csv` 로그 파일을 비교 분석하여, 현재
발생하는 통신 장애(WinError 10054 및 데이터 공백)의 근본 원인을 확정하고
실질적인 해결책을 제시합니다.

## 2. 로그 분석 결과

### 2.1 개선 전 (`system.log`)

- **주요 오류**: `[WinError 10054] 원격 호스트에 의해 강제로 끊겼습니다`
- **증상**: 주기적으로 연결이 끊기고, 파일 기록 스레드에서
  `[Errno 22] Invalid argument` 오류가 동반됨.
- **원인 추정**: 잦은 연결 끊김으로 인한 소켓 자원 불안정 및 파일 핸들링 충돌.

### 2.2 개선 후 (`system_2.log`)

- **오류 패턴 변화**: `Data Loop Error: [WinError 10054]` 오류는 여전히
  발생하나, 파일 기록 오류(`Invalid argument`)는 사라짐. 이는 코드 개선(로컬
  Fallback, 예외 격리)이 효과가 있었음을 증명함.
- **새로운 경고**: `[WARNING] Queue full! Ring Buffer dropped oldest data.`가
  대량 발생.
  - **해석**: 통신 스레드는 데이터를 계속 수집하려고 시도하지만, 처리 속도나
    기록 속도가 따라가지 못하거나, 재연결 백오프 중에 큐가 가득 차는 현상.
- **데이터 공백**: CSV 파일(`Factory_Integrated_Log`)을 확인한 결과, 오류 발생
  시점 직후 약 6초(`31~32행`) 간 데이터 값이 기록되지 않는 "Null 구간"이 명확히
  존재함.

### 2.3 `CSV` 데이터 정밀 분석 (`Factory_Integrated_Log`)

- **정상 구간**: 0.2초 간격으로 데이터가 촘촘히 기록됨 (초당 5회).
- **공백 구간**: `WinError 10054` 발생 시점과 일치하는 구간에서 타임스탬프는
  찍히지만 센서 값들이 비어 있음.
- **결론**: 소프트웨어는 **죽지 않고 살아남아** 재연결을 시도하고 있음(System
  Resiliency 확보). 단, **연결 끊김 자체**는 해결되지 않음.

## 3. 근본 원인 (Root Cause) 확정

### 3.1 0.2초 주기의 물리적 한계 (Hard Limit)

- **현상**: 0.2초(5Hz) 주기로 압출기와 PLC에 데이터를 요청함.
- **문제**:
  1. **Extruder (Split Mode)**: 초기 Block Read 실패 시 개별 요청 모드로
     전환되어, 주기당 요청 횟수가 2회 → 5회로 증가. 초당 25회의 트랜잭션은 해당
     구형 Melsec 장비의 처리 한계를 초과하여, 장비가 **Self-Protection**
     차원에서 소켓을 Reset(10054) 함.
  2. **PLC (XGT)**: 응답 패킷을 다 받기도 전에 다음 요청을 보내거나, 처리 지연
     시 타임아웃이 발생하여 연결을 끊음.

### 3.2 큐(Queue) 폭주

- 연결이 끊기고 재연결(Backoff)하는 동안, 메인 루프나 큐잉 로직에서 데이터
  처리가 지연되거나 더미 데이터가 쌓여 `Queue full` 경고가 발생함. 이는 시스템
  자원을 불필요하게 소모함.

## 4. 최종 개선 제언 (Action Plan)

현재 코드는 "통신 오류 발생 시 소프트웨어가 뻗지 않도록" 하는 데에는
성공했습니다. 이제는 **"통신 오류 발생 자체"**를 줄여야 하며, 이는 **Code
Level이 아닌 Policy Level**의 조정이 필수적입니다.

### 4.1 [필수] 수집 주기 완화 (가장 확실한 해결책)

- **현재**: 0.2초 (초당 5회)
- **변경 권고**: **0.5초 (초당 2회) ~ 1.0초 (초당 1회)**
- **기대 효과**: 트래픽 부하를 60%~80% 감소시켜 장비의 연결 끊김(Reset) 현상을
  즉각적으로 해소 예상.

### 4.2 [권장] 압출기 Block Read 최적화 환경 조성

- 코드는 이미 Block Read(병합 읽기)를 우선 시도하도록 되어 있습니다. 현장에서
  Block Read가 실패하고 Split Mode로 넘어가는 이유(네트워크 노이즈, 장비 설정)를
  점검해야 합니다.
- **장비 설정 확인**: Melsec 압출기 네트워크 설정에서 "Max Data Size"나
  "Timeout" 설정을 확인하여 대용량 패킷 전송을 허용하도록 변경.

### 4.3 [참고] H/W 인프라 점검

- 0.2초 주기를 반드시 고수해야 한다면, 노후된 스위칭 허브나 LAN 케이블을
  교체하여 패킷 손실률을 0에 가깝게 만들어야 합니다.

## 5. 결론

소프트웨어 로직 개선(재연결, 예외처리, 버퍼링)은 완료되었습니다. 현재 남은
`WinError 10054`와 `Queue Full` 문제는 소프트웨어 버그가 아닌 **물리적 통신
부하(Overload)**에 의한 증상입니다. **수집 주기를 0.5초로 변경하는 것**만으로도
대부분의 문제가 해결될 것으로 판단됩니다.
