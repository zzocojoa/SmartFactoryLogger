# 통신 연결 문제 수정 단계별 계획 보고서

## 1. 개요
본 문서는 통신 연결 끊김 보고서의 결론을 단계별 수정 계획으로 재구성한 문서입니다. 설정값은 이식된 컴퓨터의 `config.ini`를 기준으로 판단합니다.

## 2. 결론 수정안(단계별)

### 2.1 즉시 적용 권장
1) 수집 주기 0.2초 유지 전제 확정
- 주기 변경 없이 요청 최적화, 수신 안정화, 큐 처리 개선으로 부하를 낮춤.

2) 상위 루프 안전 처리
- PLC 응답이 빈 딕셔너리여도 루프가 지속되도록 안전 접근 로직 추가.
  - 예: `ls.get('Mold1')` 방식 또는 실패 시 기본 키를 포함한 딕셔너리 반환.

3) 타임아웃 상향
- 압출기 타임아웃 0.15초 → 0.5~1.0초로 상향.

4) 로그 경로 유효성 검증 및 로컬 fallback
- 이식 PC에서 `logpath` 유효성 확인 후 실패 시 `./logs`로 자동 전환.

5) 파일 기록 오류 격리/재시도
- file_writer_thread 예외 발생 시 백오프/재시도와 오류 카운트로 지속 수집 보장.


### 2.2 단기 개선
1) 수신 완전성 보장
- 응답 길이(body_len) 또는 종결 문자열 기준으로 `recv()` 루프 구현.

2) 요청 최적화
- 압출기 레지스터 요청을 가능한 범위에서 블록 읽기로 통합하고, 실패 3회 시 분할 읽기로 전환 후 300회 성공 시 통합을 1회 재시도(실패 시 600/1200회로 증가).

3) 재연결/백오프 정책
- 연속 실패 시 지수형 backoff(1, 2, 4, 8초)로 재연결 간격을 확장.

4) 큐/버퍼 처리량 개선
- Ring Buffer 용량 상향, 배치 저장, 큐 길이 모니터링으로 데이터 드롭 최소화.

5) 중복 실행 방지
- 단일 실행 락 및 시작 버튼 비활성화로 중복 수집 스레드 방지.


### 2.3 중기 개선
1) 네트워크 품질 관측
- 스위치 포트 에러율, 링크 리셋 횟수, 케이블 상태 점검.

2) 장비 통신 스펙 확인
- 제조사 권장 요청 빈도, 동시 세션 수 제한 확인 및 정책 반영.


## 2.4 현장 테스트 로그 근거
- 대상 로그: `docs/system_2.log`
- 대상 CSV: `docs/Factory_Integrated_Log_20251231_000000.csv`
- `IO Error on D0020` 시각(2025-12-31 02:15:35, 09:37:00, 09:43:04, 09:43:43, 09:57:22)과 동일 구간에서 압출기/PLC 컬럼이 연속 31~32행 공백(약 6.2초)으로 기록됨.
- 09:57:22 구간은 30행이 모든 컬럼(Spot 포함) 공백으로 기록됨.
- 타임스탬프 간격은 연속이며 행 누락은 없고, 오류 구간만 값이 빈 형태로 남음.


## 3. 검증 방법
- 0.2초 주기 유지 상태에서 요청 최적화/수신 루프 적용 전후 10054 발생 빈도 비교.
- system_2.log의 D0020 오류 시각과 CSV 공백 행(31~32행) 구간을 매칭하여 재현 여부를 확인.
- 수신 루프 적용 후 파싱 오류/연결 종료 감소 여부 측정.
- 현장 스위치/케이블 점검 전/후 재발률 비교.
- logpath 유효성/로컬 fallback 적용 후 CSV 오픈 실패 및 Invalid argument 0건 확인.
- Queue full 경고 건수(30,360 기준)가 유의미하게 감소했는지 비교.
- 동일 시각 중복 시작 로그가 재발하지 않는지 확인.

## 4. 성능 측정 지표 및 실측 계획(0.2초 고정 전제)
### 4.1 지표 정의
1) 통신 안정성
- 장비별 통신 실패율: `WinError 10054`/총 요청 수.
- 재연결 지표: 연속 실패 횟수, 평균 복구 시간.

2) 데이터 유실
- Queue full 발생 건수 및 데이터 드롭 추정량.
- `IO Error on D0020` 시 공백 행 길이(약 31~32행) 및 발생 빈도.
- 수집 누락율: CSV 행 수 vs 기대 행 수(주기 기준).

3) 파일 기록 품질
- `Invalid argument` 및 CSV 오픈 실패 건수.
- 로그 파일 생성/롤링 성공률.

4) 처리 지연/부하
- 실제 루프 주기 편차(0.2초 기준) 및 지연 누적 시간.
- CPU/메모리 사용률(수집 스레드 중심).

### 4.2 실측 계획
1) 기준선 수집
- 변경 적용 전 24~48시간 운영 로그 수집(system.log, CSV, OS 성능 지표).

2) 변경 적용 후 재측정
- 동일 조건(0.2초, 동일 라인/네트워크)에서 24~48시간 재측정.

3) 비교/판정
- 지표별 전후 비교표 작성(건수/비율/추세).
- 목표값: 10054 0~1회/주, Queue full 현저 감소, 파일 기록 오류 0건, 중복 시작 0건.

4) 운영 반영
- 목표 미달 항목은 원인 분해(네트워크/장비 정책/클라이언트 처리) 후 보완안 재설계.
