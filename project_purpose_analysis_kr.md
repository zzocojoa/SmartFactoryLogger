# 프로젝트 목적 및 실무적 분석 보고서

## 1. 프로젝트 개요 (Overview)

본 프로젝트(`SmartFactoryLogger`)는 **알루미늄 압출 공정(Aluminum Extrusion
Process)** 현장에서 발생하는 다양한 설비 데이터를 실시간으로 수집, 통합,
시각화하고 로그를 기록하는 **현장 데이터 수집 및 모니터링 시스템(DAS, Data
Acquisition System)**입니다.

서로 다른 통신 프로토콜을 사용하는 이종 장비(Extruder, PLC, Pyrometer)를 하나의
시스템으로 통합하여, 공정 변수를 실시간으로 감시하고 품질 관리에 필요한 이력
데이터(Traceability Data)를 확보하는 것을 주 목적으로 합니다.

## 2. 주요 기능 및 구성 (Key Features & Components)

### 2.1 이종 설비 프로토콜 통합 (Multi-Protocol Integration)

본 시스템은 제조사나 모델이 다른 세 가지 핵심 설비와 통신하여 데이터를
수집합니다.

- **압출기 (Extruder)**: TCP Socket 기반의 커스텀
  프로토콜(`Modules/extruder.py`)을 사용하여 압력, 전/후방 온도, 램 속도(Ram
  Speed), 빌렛 길이, 생산 카운트 등을 수집합니다.
- **LS PLC (XGT 계열)**: LS산전의 LSIS-XGT 프로토콜(`Modules/ls_plc.py`)을 통해
  금형(Mold) 온도(6개 존), 빌렛(Billet) 예열 온도 등 보조 설비 데이터를
  수집합니다.
- **SPOT 비접촉 온도계 (Pyrometer)**: Ametek Land SPOT 장비와 연동하여 압출되는
  제품의 실제 표면 온도를 비접촉으로 측정합니다.

### 2.2 실시간 데이터 처리 아키텍처 (Concurrency & Architecture)

- **Producer-Consumer 패턴**: `main.py`는 데이터 수집(Producer), 파일
  로깅(Consumer 1), GUI 표시(Consumer 2)로 역할을 명확히 분리하여
  설계되었습니다.
- **멀티스레딩 & 병렬 처리**: `ThreadPoolExecutor`를 사용하여 3개의 장비에서
  동시에 데이터를 요청하므로, 특정 장비의 응답 지연이 전체 사이클 타임(0.2초)에
  영향을 주지 않도록 설계되었습니다.
- **Queue 기반 통신**: 수집된 데이터는 `queue.Queue`를 통해 스레드 간 안전하게
  전달되어 데이터 손실(Data Loss)을 방지합니다.

### 2.3 데이터 로깅 및 품질 관리 (Logging & QoS)

- **정밀 로깅**: 0.2초 단위의 고해상도 데이터를 CSV로 저장하여, 불량 발생 시점의
  공정 조건(속도, 온도, 압력 등)을 역추적(Traceability)할 수 있습니다.
- **가시성 확보 (GUI)**: Tkinter 기반의 GUI를 통해 현장 작업자가 직관적으로 공정
  상태를 파악할 수 있도록 숫자를 시각화하고 그래프를 제공합니다.

## 3. 실무적/비즈니스 가치 (Business Value)

1. **품질 보증 및 불량 분석 역량 강화**:
   - 압출 품질에 치명적인 영향을 주는 **금형 온도(Mold Temp)**와 **빌렛
     온도(Billet Temp)**, **압출 속도(Speed)** 간의 상관관계를 분석할 수 있는
     기반 데이터를 제공합니다.
   - 단순 생산 수량 체크를 넘어, **"어떤 조건에서 생산된 제품인가?"**에 대한
     이력을 남김으로써 납품 품질 신뢰도를 높입니다.

2. **레거시 설비의 디지털 전환 (Digital Transformation)**:
   - 일체형 패키지 소프트웨어가 아닌, 현장의 구형 압출기와 최신 센서(SPOT)를
     커스텀 소프트웨어로 연결함으로써 저비용으로 **스마트 팩토리의 기초
     단계(데이터 수집)**를 구축했습니다.

3. **운영 효율성 증대**:
   - 작업자가 일일이 설비 계기판을 확인하러 다니지 않고, 중앙 모니터링 PC에서
     전체 상황을 파악하여 이상 발생 시 즉각 대응이 가능합니다.

## 4. 종합 의견 (Conclusion)

이 프로젝트는 단순한 로거(Logger)를 넘어, **제조 현장의 핵심 KPI(생산성,
품질)**를 관리하기 위한 필수 미들웨어 역할을 수행합니다. 실무적으로는 향후
**MES(생산관리시스템)와의 연동**이나 **예지 보전(Predictive Maintenance)** 모델
개발을 위한 학습 데이터셋 구축의 전초 기지로서 매우 높은 가치를 가집니다.

## 5. 잠재적 문제점 및 개선 제안 (Potential Issues & Improvements)

전문 개발자 관점에서 코드 검토 결과, 운영 환경에서 발생할 수 있는 잠재적 위험
요소와 개선이 필요한 항목들이 발견되었습니다.

### 5.1 아키텍처 및 통신 안정성 (Architecture & Reliability)

1. **예외 처리 미흡 (Silent Failures)**:
   - `extruder.py`와 `ls_plc.py`에서 `try-except` 구문이 사용되고 있으나, 예외
     발생 시 구체적인 로그를 남기지 않고 단순히 `pass` 하거나 `None`을 반환하는
     경우가 많습니다. 이는 통신 장애 원인(Timeout, Connection Refused 등)을
     파악하기 어렵게 만듭니다.
   - **제안**: `logging` 모듈을 각 모듈에 적용하여 에러 스택 트레이스를 파일로
     남기도록 개선해야 합니다.

2. **패킷 처리의 유연성 부족 (Fragile Binary Parsing)**:
   - `ls_plc.py`에서 응답 패킷을 파싱할 때 고정된 오프셋(`data[30:]`)을 사용하고
     있습니다. PLC 설정 변경이나 응답 포맷이 미세하게 바뀌면 파싱 에러가 발생할
     위험이 큽니다.
   - **제안**: `pymodbus`와 같은 검증된 표준 라이브러리를 사용하거나, 응답
     헤더를 동적으로 분석하는 로직으로 강화해야 합니다.

3. **데이터 손실 가능성 (Queue Overflow)**:
   - `main.py`에서 `log_queue`와 `gui_queue`가 꽉 찼을 때(`queue.Full`), 단순히
     경고(`queue full!`)만 출력하고 데이터를 **버리도록(Drop)** 구현되어
     있습니다. 디스크 쓰기 속도가 느려지거나 GUI가 멈추면 중요 데이터가 유실될
     수 있습니다.
   - **제안**: 큐 사이즈를 늘리거나, Ring Buffer 구조를 도입하여 오래된
     데이터부터 버리되, 데이터 유실 발생 사실을 운영자에게 강력하게 알리는
     경보(Alarm) 시스템이 필요합니다.

### 5.2 유지보수 및 확장성 (Maintainability)

1. **하드코딩된 설정 및 의존성**:
   - PLC 메모리 주소(`%dw...`)나 특정 비트 위치가 코드 내(또는 `config.ini`의
     단순 문자열)에 흩어져 있어, 설비 변경 시 수정이 어렵습니다.
   - **제안**: 태그 매핑 테이블(Tag Mapping Table)을 별도 JSON/YAML 파일로
     분리하고, 주소 변경 시 코드 수정 없이 설정 파일만 교체할 수 있도록
     추상화해야 합니다.

2. **테스트 코드 부재 (Lack of Tests)**:
   - 단위 테스트(Unit Test)가 없어 로직 변경 시 기존 기능이 정상 작동하는지
     검증하기 어렵습니다. 특히 프로토콜 파싱 로직은 모의 객체(Mock)를 활용한
     테스트가 필수적입니다.

### 5.3 성능 및 리소스 관리 (Performance)

1. **파일 I/O 확장성 문제**:
   - 현재 CSV 파일 하나에 모든 데이터를 기록하고 있습니다. 수집 주기가 0.2초로
     매우 빠르기 때문에 파일 크기가 급격히 커질 수 있으며, 장기간 운영 시 파일을
     여는 속도가 느려질 수 있습니다.
   - **제안**: 날짜/시간별로 파일을 자동 분할(Rotation)하는 로직은 이미 있으나,
     데이터베이스(SQLite/PostgreSQL) 도입을 검토하여 검색 성능을 높이는 것이
     좋습니다.
